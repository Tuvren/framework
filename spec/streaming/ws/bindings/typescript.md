# TypeScript Binding — Event Stream WebSocket Transport (`tuvren.framework.event-stream-ws`)

Binding projection: `typescript/streaming/ws` (`@tuvren/stream-ws`).

## Surface

| Authority model | TypeScript projection |
| --- | --- |
| `WsHandshakeRequest` / `WsHandshakeAck` | in-band handshake exchange consumed by the transport's socket seam |
| `WsOutboundFrameEnvelope` / `WsInboundFrameEnvelope` | `{ kind: "frame", cursor?, frame }` carriage envelopes wrapping `SessionOutboundFrame` / `SessionInboundFrame` (packet `tuvren.framework.host-session`) |
| `WsPing` / `WsPong` | in-band heartbeat messages |
| `WsCloseCode` | `WsCloseCode` union and `WS_CLOSE_CODE_*` constants |

Exports (all `@experimental`, ADR-056 posture):

- `createWsSessionTransport(...)` — adapts a host's `RemoteClientSession` (packet `tuvren.framework.remote-session`, ADR-063) and a `WsSocketSink` into the handshake/frame/inbound-routing state machine described by ADR-062 as amended by ADR-063. This is **pure carriage**: the transport owns only the WebSocket-level concerns — handshake (including all four handshake close-code failure paths and serialized async `authorize`), non-event/event frame envelope carriage, `ping`/`pong` echo, the settled unknown-kind inbound-routing policy below, application-level heartbeat/half-open detection (`4004`), and bounded outbound backpressure (`4005`). It does **not** claim a `DuplexSessionBinding.outbound()` stream, create a sequencer, or record a replay buffer — `options.session` owns all of that, for the session's whole life, across however many transport instances attach and detach beneath it in turn.
- `WsSocketSink` — the runtime-agnostic push-model socket seam (`send(data)`, `close(code, reason?)`, optional `bufferedAmount()`) a host adapts from `Bun.serve` websockets, the browser `WebSocket` global, Node `ws`, or an in-memory pair in tests. `bufferedAmount()` is optional and mirrors the WebSocket `bufferedAmount` concept (bytes queued on the real socket); when a host's sink omits it, backpressure enforcement is disabled even if the `backpressure` option is set.
- `parseWsMessage(data)` and the `WsHandshakeRequest` / `WsHandshakeAck` / `WsResumeStatus` / `WsOutboundFrameEnvelope` / `WsInboundFrameEnvelope` / `WsPing` / `WsPong` wire types — the pure, throw-free structural parser and its message shapes. Unchanged by ADR-063: no wire schema moved.
- `WS_CLOSE_CODE_HANDSHAKE_INVALID` (`4000`), `WS_CLOSE_CODE_PROTOCOL_VERSION_UNSUPPORTED` (`4001`), `WS_CLOSE_CODE_SESSION_NOT_FOUND` (`4002`), `WS_CLOSE_CODE_AUTH_REJECTED` (`4003`), `WS_CLOSE_CODE_HEARTBEAT_TIMEOUT` (`4004`), `WS_CLOSE_CODE_BACKPRESSURE_EXCEEDED` (`4005`), and the `WsCloseCode` union type; the six close codes and their semantics are fixed by ADR-062 §5, with two additional reuses recorded below for the session-composition failure modes ADR-063 introduces.
- `WsSessionTransportOptions.heartbeat?: { intervalMs, timeoutMs }` — enables heartbeat/half-open detection. Once the handshake completes, the transport sends `{kind: "ping"}` every `intervalMs` and arms a `timeoutMs` timer after each ping; any inbound message (not only `pong`) clears that timer, since a peer actively sending frames is demonstrably alive — `pong` remains the answer a quiet peer produces, because the transport always answers an inbound `ping` with `pong`. A timer that fires with no inbound activity reports warning code `ws_transport_heartbeat_timeout` and closes `4004`. Omitting `heartbeat` runs no timers. Throws `RangeError` at construction for a non-positive or non-finite `intervalMs`/`timeoutMs`; `timeoutMs` may be less than, equal to, or greater than `intervalMs`.
- `WsSessionTransportOptions.backpressure?: { maxBufferedBytes }` — enables bounded outbound backpressure. Enforced only when both this option and the sink's `bufferedAmount()` capability are present: before each outbound send (live sends, and the replay/redelivery burst flushed right after the handshake ack), the transport checks `sink.bufferedAmount()` against `maxBufferedBytes`; on overflow it reports warning code `ws_transport_backpressure_exceeded` and closes `4005` instead of sending — never a silent drop. Throws `RangeError` at construction for a non-positive or non-finite `maxBufferedBytes`.

## ADR-063 composition: `binding → session → transport → socket`

`@tuvren/stream-ws` no longer takes a `DuplexSessionBinding` or a replay buffer directly. A host composes, in order:

1. A `DuplexSessionBinding` (packet `tuvren.framework.host-session`) over an `ExecutionHandle`.
2. A `RemoteClientSession` (packet `tuvren.framework.remote-session`, `@tuvren/remote-session`) over that binding — this is what claims the binding's single `outbound()` stream (lazily, on its own first `attach`), owns the one sequencer and one replay buffer for the session's whole life, and redelivers unanswered `client_invocation` frames on reattach.
3. A `WsSessionTransport` (this surface) over that session — `createWsSessionTransport({ session, sink, ... })`.
4. The host's real socket, adapted to `WsSocketSink`.

The same `RemoteClientSession` instance may be attached by a sequence of `WsSessionTransport` instances over its life: when a socket drops, the host calls `transport.close()` (or the transport closes itself via heartbeat/backpressure policy), which detaches the sink from the session — starting the session's own `disconnectGraceMs` window — without ending the session. A later transport instance, constructed over the SAME session, resumes by handshaking with the session's opaque cursor; the session replays retained `event` frames after that cursor and redelivers any `client_invocation` still unanswered, both before the new transport's live forwarding resumes. Sequence numbering never restarts across this reattach, because there is only ever one sequencer, owned by the session — this transport creates none.

**Host-composition responsibility carried over from the transport:** because the session (not this transport) owns the pump driving `binding.outbound()`, this transport has no intrinsic signal for "the underlying turn ended, close the socket normally". That signal is `RemoteClientSessionOptions.onEnded`, and it is the host's job to wire it to whichever transport is currently attached (e.g. `onEnded: (reason) => currentTransport?.close(1000, reason)`), tracking "currently attached" itself. This differs from pre-ADR-063 `@tuvren/stream-ws`, where the transport's own outbound pump completing closed the socket with `1000` unconditionally.

## Transport rules (normative for hosts, from `spec/streaming/ws/typespec/main.tsp`)

1. **Handshake-first.** The first message a client sends after the socket opens MUST be a schema-valid `WsHandshakeRequest`; the first message a server sends back MUST be `WsHandshakeAck`. The session protocol is carried only after a successful handshake exchange, which is also the point at which this transport attaches its sink to `options.session` (see composition above).
2. **Cursor only on `kind: "event"` outbound envelopes.** `WsOutboundFrameEnvelope.cursor` (the opaque ADR-061 resume token) is present exactly when the wrapped session frame has `kind: "event"` — the canonical event stream is the only sequenced, replayable surface. `client_invocation` and `session_rejection` frames carry no cursor.
3. **Frame-level problems surface as `session_rejection` frames, never socket closes.** A schema-invalid or state-refused inbound frame is the session binding's concern, not the transport's; it is answered with an ADR-060 `session_rejection` frame on the still-open socket. **Settled unknown-kind policy:** any post-handshake message that is not a recognized envelope kind — malformed JSON, an object with no `kind`, an unrecognized `kind`, or a stray `handshake`/`handshake_ack` arriving after the handshake completed — is routed to `session.dispatchInbound()` exactly like a `frame` envelope's inner payload, carrying whatever value `parseWsMessage` produced (the parsed object, or the raw decoded text when `JSON.parse` itself failed). The session's own structural validation (delegating to the underlying binding) turns that into a `session_rejection` frame with code `session_frame_invalid`. The socket never closes for a frame-level or unrecognized-message problem — only handshake failures and this transport's own heartbeat/backpressure policy close the connection.
4. **Close codes are reserved for connection-level conditions**: `4000` handshake_invalid, `4001` protocol_version_unsupported, `4002` session_not_found, `4003` auth_rejected, `4004` heartbeat_timeout, `4005` backpressure_exceeded, and `1000` for normal end-of-session close. ADR-063 adds two documented reuses rather than new codes:
   - **`4000` (handshake_invalid) also covers a double-attach refusal.** A second concurrent handshake for a session that already has a live sink attached causes `session.attach()` to throw; this transport closes the **new** socket (never the already-attached one) with `4000` — the closest existing code to "this handshake cannot be honored as presented" — and leaves the first connection untouched.
   - **`4002` (session_not_found) also covers a handshake against a permanently-ended session.** If `session.attach()` throws because the session already ended (explicit `close()` or disconnect-grace expiry), this transport closes with `4002` — there is no live session left for the presented identity to bind to.

   `RemoteClientSession.attach` throws a `TuvrenRuntimeError` carrying a stable code for each failure — `remote_session_already_attached` (exported as `REMOTE_SESSION_ALREADY_ATTACHED`) for a live second attach and `remote_session_ended` (`REMOTE_SESSION_ENDED`) for a permanently-ended session — and this transport branches on `error.code`, never on message text, per the core error contract. An unrecognized attach failure is treated as `4002`: refusing the handshake is the invariant; the code split only decides which connection-level diagnosis the refused peer receives.
5. **Ping/pong liveness.** Either side MAY send `{kind: "ping"}`; the receiver MUST answer `{kind: "pong"}`. When `options.heartbeat` is configured, any inbound message counts as liveness (not only `pong`), and a timer that fires with no inbound activity within the configured `timeoutMs` since its ping was sent is half-open detection and closes with `4004`. Heartbeat timers run only when `options.heartbeat` is supplied; omitting it disables timeout enforcement entirely (the transport still echoes `pong` for an inbound `ping`).
6. **Bounded outbound buffer, close-on-overflow.** The outbound bound is a configurable byte budget (`options.backpressure.maxBufferedBytes`) enforced against the sink's optional `bufferedAmount()` report before each outbound send (live sends, and the replay/redelivery burst flushed immediately after the handshake ack). Enforcement requires both the option and the sink capability; a sink that cannot report buffering runs without transport-level backpressure enforcement. When enforced, overflow closes with `4005` rather than silently dropping a frame — a silent drop would create a sequence gap the resume cursor could neither explain nor repair; a close converts overflow into an honest reconnect-with-cursor. **No silent drop is permitted.** Unlike the pre-ADR-063 transport, this is now unconditionally safe for `event` frames regardless of *this* transport's state: `RemoteClientSession` records every sequenced event into its replay buffer as part of forwarding it to whichever sink is attached, so a frame caught by an overflow close is still recoverable by the next reattach's cursor replay even though this transport itself keeps no buffer.
7. **Overflow and a dropped link only detach; they never end the session.** On any close this transport initiates — heartbeat half-open, backpressure overflow, or an explicit host-driven `transport.close()` — the transport calls `session.detach(reason)` (only if its own handshake had actually attached a sink; a handshake that never got that far has nothing to detach). It never calls anything that ends or permanently closes the session. Whether a later reattach is still possible is entirely the session's `disconnectGraceMs` window, not a transport-level decision.

## Layering rule

- Frame semantics (the `SessionOutboundFrame` / `SessionInboundFrame` vocabulary itself, including `session_rejection`) are owned by `spec/host/session/` (packet `tuvren.framework.host-session`), not this surface.
- Cursor semantics (encoding, replay-window resolution, `ReplayStatus`) are owned by `spec/streaming/resume/` (packet `tuvren.framework.event-stream-resume`), not this surface.
- Session lifecycle semantics (sequencer/replay-buffer ownership, reattach, unanswered-`client_invocation` redelivery, disconnect grace, dispatch timeout) are owned by the `tuvren.framework.remote-session` packet (ADR-063, `@tuvren/remote-session`'s own authority), not this surface.
- This surface (`spec/streaming/ws/`, packet `tuvren.framework.event-stream-ws`) owns **carriage only**: handshake, heartbeat, close codes, and outbound queueing. No frame, cursor, or session-lifecycle semantic may drift into the transport without violating this packet's `forbiddenAuthoritySources`.

## `RemoteClientSession` integration obligations

Per `typescript/host/remote-session/src/lib/remote-client-session.ts` and its documentation, a transport built on top of `RemoteClientSession` MUST honor:

- **At most one sink attached at a time.** `session.attach(sink, {cursor?})` throws if a sink is already attached; a transport MUST NOT retry or paper over that throw by displacing the existing sink — the correct response is to close the *new* connection (see close-code rule 4 above) and leave the existing one alone.
- **`session.dispatchInbound` calls are wrapped in the transport's own `try`/`catch`.** The session does not protect callers from adapter-level failures (malformed socket payloads, sink errors); the transport is responsible for isolating each `dispatchInbound` invocation so a single bad inbound message cannot take down the connection's read loop.
- **Detach, never end, on a transport-initiated close.** `session.close()` is a host-level, cross-transport decision (ending the session permanently); a single transport's own closure — for any reason — calls `session.detach(reason)` only.
