### ADR-056 TSDoc `@experimental` Is the Canonical Experimental-Surface Marker, Enforced by the API-Snapshot Gate

- **Status:** accepted
- **Context:** ADR-054 froze the stable core (`@tuvren/core` primitives, the Durable-Read Surface, `ExecutionHandle`/`awaitResult`, `createTuvren`, and the published leaf packages) and carved out the still-settling `@tuvren/core/capabilities` advanced classes as an explicitly excluded, experimental surface — but deferred "the exact ... experimental-marker mechanism" to "the SaaS-readiness freeze epic." This ADR is that finalization, for PRD CAP-P0-070, whose capability text requires "any not-yet-stable surface [be] explicitly separated and marked experimental so adopters can depend on the stable core without absorbing churn." The 2026-07-04 audit (`grep -rn "@experimental" typescript/core/src`) found zero markers exist today: `@tuvren/core/capabilities` (`typescript/core/src/capabilities/index.ts`) is presently a ~43-line type-only re-export barrel (per ADR-046 and the phased realization of ADR-047) with no stability annotation on any of its re-exported types. Two constraints must be resolved together, not sequentially: (1) the marker must be human-visible in editor tooling and generated docs, and (2) the forthcoming API-surface snapshot/diff gate must programmatically distinguish a stable-core breaking change (blocked) from an experimental-surface change (permitted) — a marker chosen without the gate's read model in mind risks a mismatch discovered only after tooling is built.
- **Decision:** TSDoc `@experimental` release tags on individual exports are the canonical experimental-surface marker for the stable-core freeze, chosen over a dedicated experimental subpath (e.g. `@tuvren/core/capabilities/experimental`) or a package-level-only notice.
  1. **Why a tag on the export, not a subpath split.** A subpath boundary is import-path-shaped: graduating one export to stable would force every adopter to change an import path, which is itself a breaking change at the exact moment the export becomes safe to depend on — precisely the churn CAP-P0-070 asks the marker to prevent. A TSDoc tag graduates an export by deleting a comment: the import path (`@tuvren/core/capabilities`) never moves, and removing `@experimental` from a previously-tagged export is additive (semver-minor), not breaking. The subpath already exists as the coarse capability-orchestration boundary from ADR-046; this decision does not disturb it and instead operates one level finer, at the individual export.
  2. **Why TSDoc specifically, not a bespoke comment convention or a runtime flag.** TSDoc release tags (`@alpha`/`@beta`/`@experimental`/`@public`) are a standardized, tool-recognized vocabulary that API-surface extraction tooling (an api-extractor-class tool, or an equivalent tsc `.d.ts`/type-checker-driven walker selected in the freeze-gate execution work) reads natively when building a public API model. Adopting the standard vocabulary means the freeze gate's stable/experimental classification is derived mechanically from the same annotation a developer sees in-editor and in generated docs, rather than maintained as a second, hand-kept list that can drift from the source. A bespoke convention or a runtime-only flag would not be visible to static extraction tooling and would require the gate to re-invent tag semantics; a runtime flag additionally cannot express "this exported type is experimental" for type-only exports, which is most of `@tuvren/core/capabilities` today.
  3. **Marking mechanics.**
     - Every current export of `@tuvren/core/capabilities` receives a `/** @experimental */` TSDoc tag at initial freeze. The subpath's package-level documentation additionally carries a subpath-level experimental notice so the boundary is visible without reading individual export docs.
     - The freeze/diff gate treats tags as the sole source of stable/experimental classification:
       ```
       stable snapshot   = frozen public surface \ { exports tagged @experimental }
       diff on new run:
         - export gains @experimental (was untagged/stable)  => BLOCKED (removes a stable guarantee)
         - export loses @experimental (was tagged)            => ALLOWED, recorded as semver-minor
         - signature change on an @experimental export        => ALLOWED, not gated
         - signature change on an untagged (stable) export    => BLOCKED unless semver-major
       ```
     - **Consistency floor:** if the authority (this ADR, or a successor capability-orchestration ADR) declares a subpath wholly experimental, the gate fails the build when any export under that subpath lacks the `@experimental` tag — an untagged export under a declared-experimental subpath is a documentation defect, not a silent stable promotion.
     - **Graduation rule:** an export only becomes part of the enforced stable snapshot once it appears untagged in a snapshot run; there is no separate "promote" ceremony beyond removing the tag and letting the gate's next run absorb it.
     - Marker declarations (which exports/subpaths carry `@experimental` and why) are recorded in the core authority packet's surface listing, so conformance plans, generated docs, and the gate's expectations stay aligned with a single authored source rather than three independently maintained lists.
- **Consequences:** Adopters can depend on `@tuvren/core/capabilities` today for the advanced capability-orchestration classes and keep their import paths stable through graduation — only the tag disappears, which is additive. The freeze gate requires a TSDoc-aware surface extractor; the concrete tool (api-extractor-class vs. a custom tsc/`.d.ts` walker) is a tooling choice left to the execution plan, not fixed here. Linting/CI must enforce the consistency floor so an untagged export cannot silently ride into the stable snapshot under a declared-experimental subpath. Generated documentation must surface the experimental badge per export so adopters see the marker in the same place they see the type signature. This decision does not reopen ADR-046/ADR-047's phased realization of the capability-orchestration classes; it only fixes how their in-flux status is expressed and mechanically enforced once the stable-core freeze (ADR-054) takes effect.
