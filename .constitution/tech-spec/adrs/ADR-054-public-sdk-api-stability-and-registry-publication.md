### ADR-054 Public SDK API Stability and Registry Publication, Sequenced After Tenancy and Data Lifecycle

- **Status:** accepted
- **Context:** PRD v0.10.0 CAP-P0-070 promotes SDK API-stability and package publication from the deferred post-trust roadmap to active P0: a host cannot build a production product on an unpublished, unversioned SDK, and first-party hosts (e.g. airsheet) need the runtime as a released dependency. The curated surface already exists — `@tuvren/core` (subpath-exported primitives) plus `@tuvren/runtime` (`createTuvren` and curated re-exports), with leaf packages peer-depending on `@tuvren/core` (ADR-037, ADR-040). Two risks shape this decision: freezing a surface that later needs a scope or erasure parameter (Architecture freeze-before-scope failure class), and freezing the still-settling capability-orchestration advanced classes.
- **Decision:**
  1. **Sequencing (resolved fork Q3).** The freeze and first registry publication happen **after** the tenancy/isolation work (ADR-048/049) and the data-lifecycle/erasure work (ADR-051) land, so the stable public contract already accounts for scope binding (construction-time, so signatures are unaffected — ADR-048) and for the shreddable-payload/erased-read shapes (ADR-051) rather than breaking to admit them.
  2. **Stable core vs. experimental surface.** The frozen, semver-guaranteed surface is the stable core: `@tuvren/core` primitives (messages, tools, events, errors, execution, driver/provider contracts, extensions), the Durable-Read Surface, the `ExecutionHandle`/`awaitResult` surface, `createTuvren`, and the published leaf packages (backends, stream adapters, ReAct driver, AI SDK bridge, MCP client). Surfaces still in flux — notably the deeper capability-orchestration classes under `@tuvren/core/capabilities` — are explicitly marked experimental and excluded from the stability guarantee, so adopters pin the stable core without absorbing churn.
  3. **Publication mechanics.** Packages publish to the public registry under semantic versioning, with `@tuvren/core` as the peer-depended shared primitive (version-skew safety per ADR-037). The experimental subpath carries an explicit experimental marker in its types and docs.
- **Consequences:** External adoption and first-party dogfooding become possible against a released, versioned dependency. The construction-time scope binding (ADR-048) is what makes the freeze safe without a scope parameter on read signatures. The capability-orchestration advanced classes can keep evolving behind the experimental marker until their provider-statefulness posture (ADR-053) and native-client trigger (ADR-055) settle. A breaking change to the stable core after publication is a semver-major with a migration window (consistent with ADR-037's deprecation-window precedent). Exact registry names, version policy, and the experimental-marker mechanism are finalized in the SaaS-readiness freeze epic.
