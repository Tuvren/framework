### ADR-053 Tuvren Is the Unconditional Conversation-State Owner

- **Status:** accepted
- **Context:** PRD v0.10.0 CAP-P0-069 (strengthening CAP-P1-031) and Architecture v0.10.0 (Conversation-State Ownership Model) require that the durable lineage be the unconditional source of truth for a provider request. Providers have moved from stateless to stateful execution: OpenAI's Responses API offloads conversation state server-side (chain by `previous_response_id`, `store:true` defaulting to ~30-day retention, reasoning models needing `reasoning.encrypted_content` carried across turns), while Anthropic keeps state client-side but makes resending cheap via explicit `cacheControl` ephemeral breakpoints. The baseline bridge (`@tuvren/provider-bridge-ai-sdk`, `LanguageModelV3`) round-trips these only as opaque `providerOptions`/`providerMetadata`. ADR-005 already states the AI SDK bridge is not the runtime loop or the durable-state owner; this ADR makes the conversation-state posture explicit before any native client work and before the freeze.
- **Decision:** The durable lineage is always authoritative; provider state is a reconstructable optimization, never a dependency:
  1. A provider request is always fully reconstructable from Tuvren's own lineage. The Provider Gateway must be able to build the next request from durable history without relying on any provider-held state.
  2. Provider server-side state and continuity artifacts (e.g. a response id, `encrypted_content`, thinking-continuity signatures) are carried only as reconstructable optimizations. They are stored as sensitive untrusted-edge payloads (shreddable per ADR-051), already modeled as opaque `providerMetadata` continuity tokens in `docs/KrakenFrameworkSpecification.md` §3.x, and never required for correctness. The default posture prefers not delegating authority to provider-managed stores (e.g. prefer carrying encrypted reasoning over depending on a provider's server-side `store`).
  3. Provider-side caching (OpenAI implicit, Anthropic explicit `cacheControl`) is correctness-neutral: a cache miss changes cost, never outcome. Cache breakpoint placement is an optimization the runtime may drive from its context-manifest knowledge, but it never changes the reconstructable request.
- **Consequences:** Durability, recovery, and provider neutrality stay decoupled from any provider's retention window or availability, preserving the single-source-of-execution-truth principle as providers evolve. This is a posture/extension of ADR-005, not a superseding: the baseline bridge remains, and the bridge must extract continuity artifacts into lineage and re-inject them rather than letting the provider hold authoritative state. It also bounds a tenancy/erasure risk: relying on a provider's multi-day server-side store would put tenant data outside the runtime's erasure reach, which this posture avoids. `docs/KrakenFrameworkSpecification.md` is amended with the conversation-state-ownership rule. The accepted cost — full input tokens per turn (no server-side context offload) — is the eventual business case for native provider clients (ADR-055), which can place cache breakpoints using manifest knowledge the bridge cannot express.
