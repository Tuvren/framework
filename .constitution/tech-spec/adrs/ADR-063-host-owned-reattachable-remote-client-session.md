### ADR-063 Host-Owned Reattachable Remote Client Session: One Sequencer, One Replay Window, Redelivered Invocations

- **Status:** accepted.
- **Context:**
  - ADR-060 fixed the duplex session frame vocabulary and shipped `@tuvren/host-session`; ADR-061 added wire-level sequencing, the opaque resume cursor, and a bounded replay window in `@tuvren/stream-core`; ADR-062 added `@tuvren/stream-ws` carriage over a host-adapted `WsSocketSink`. The protocol and the carriage exist. Nothing in the repository has ever run them over a real socket: there is no WebSocket runtime dependency, no listening server, and every `AttachedClientEndpoint` and every `WsSocketSink` in the tree is an in-process fake. Issue #102 names the gap, and the semantics most likely to harbour latent defects — late results across reconnects, lease staleness after process death, re-dispatch dedup on recovery — are precisely the ones only a network implementation exercises.
  - ADR-062 §6 recorded two obligations against this issue rather than pretending the cursor covered them. First, `@tuvren/stream-ws` creates its sequencer **per connection** while `DuplexSessionBinding.outbound()` is single-claim (a second claim throws `duplex_session_outbound_already_claimed`), so one binding can only ever be carried by one socket for its lifetime and cross-socket live continuation is undefined; the ADR states that sequencer and replay-buffer ownership must move host-side "so that a shared replay window is never fed by two independently numbered sequencers." Second, a non-event outbound frame (`client_invocation`, `session_rejection`) lost to a connection-level close is **not replayable** — the cursor addresses only the sequenced event stream — and its loss is visible to the host solely as a session that never receives the awaited `client_result`, i.e. as a hung tool call.
  - Issue #102 additionally asks for a decision on lease-counter durability across process restarts (`ClientEndpointBoundary`'s counter is in-memory and "resets to 0 after a process restart with a fresh boundary"), and for network endpoint policy — dispatch timeout and detach-on-disconnect — that ADR-060 explicitly left to this issue.
- **Decision:**
  1. **Session lifecycle becomes a distinct seam, above the frame binding and below carriage.** A new experimental package `typescript/host/remote-session` (`@tuvren/remote-session`) depends on `@tuvren/host-session` and `@tuvren/stream-core` and owns exactly one concern: keeping a duplex session alive across a link that can drop. It is transport-agnostic — nothing in it names WebSocket — so an SSE-plus-inbound-channel host, an IPC host, or a test harness composes the same object. The alternative of growing `@tuvren/host-session` was rejected: it would give the frame-vocabulary binding a streaming dependency and three reasons to change, the concern-drift ADR-062's consequences section explicitly guards against.
  2. **`RemoteClientSession` holds the single `outbound()` claim for the session's whole life.** It claims `binding.outbound()` exactly once at construction-of-first-attach, wraps that stream with **one** `createSequencedTuvrenStreamEvents` instance, and records every sequenced frame into **one** `createReplayBuffer`. Transports no longer consume the binding; they attach beneath the session via `attach(sink)` / `detach(reason)`, at most one live sink at a time, and a second concurrent `attach` is a programming error rather than a silent second consumer. This is the ownership move ADR-062 §6 requires, and it makes cross-socket continuation well-defined by construction: sequence numbering never restarts, because there is only ever one sequencer, so a replay window can never be fed by two independent numberings.
  3. **Unanswered `client_invocation` frames are recovered by redelivery, not by replay.** The session tracks every dispatched `client_invocation` whose `callId` has not yet been settled by a `client_result`. On reattach it re-sends those frames — after cursor replay of the sequenced event stream, before live forwarding. Redelivery is safe *because the correlation handles already in the wire vocabulary make it safe*, not by adding a delivery-guarantee layer: the session-level pending-call table settles a `callId` exactly once (a second result for a settled call is already `capability_result_stale` per ADR-060), the boundary's `leaseToken`/`callId` echo check independently discards results produced for a superseded dispatch, and `ClientInvocationEnvelope.idempotencyKey` (ADR-052, `sha256(runId, callId, fencingToken)`) lets the peer suppress a duplicate *side effect* rather than merely a duplicate message. `session_rejection` frames are deliberately **not** redelivered: they are advisory responses to an inbound frame the peer can simply re-send, and re-emitting them would surface rejections for frames the peer has already abandoned. Ordering between the sequenced event channel and the unsequenced `client_invocation` channel is not guaranteed across a reattach; peers correlate by `callId` and must not infer causality from interleaving.
  4. **Disconnect is a grace window, not an immediate failure.** `detach` starts a `disconnectGraceMs` timer instead of failing in-flight dispatches. A reattach inside the window cancels the timer and redelivers per §3; expiry settles every pending dispatch with `capability_binding_unavailable` and detaches the endpoint from the `ClientEndpointBoundary`, so subsequent invocations are refused rather than dispatched into a dead link. This is the "detach-on-disconnect" policy issue #102 scopes, expressed so that the common case — a tab reload, a laptop wake, a mobile network handover — costs a replay rather than a failed turn. A grace window of zero reproduces immediate-detach semantics for hosts that want them.
  5. **Dispatch timeout is independent of connection health,** because a connected peer that never answers is a distinct failure from a dropped link. `dispatchTimeoutMs` bounds how long the runtime waits for any one `client_result`; expiry settles that dispatch with a new stable code `capability_dispatch_timeout` (`typescript/core/src/lib/capability-error-codes.ts`), joining `capability_binding_unavailable` and `capability_result_stale` in the §4.21 error family. A distinct code rather than reuse is deliberate: "no endpoint attached", "the endpoint answered for a superseded dispatch", and "the endpoint accepted the work and went quiet" are three different operator diagnoses and three different host remediations. The timeout is also why the session synthesizes a well-shaped `{ code, error }` error result rather than throwing: a thrown rejection would take the lower-fidelity code-less path `spec/host/client-endpoint-integration.md` already warns hosts away from.
  6. **The lease counter stays process-local, and the analysis is written down rather than left as an inline caveat.** `ClientEndpointBoundary`'s counter is not made durable. Its purpose is intra-process disambiguation of repeated dispatches of the same `callId` within one boundary instance; cross-invocation uniqueness comes from `callId`, which the runtime mints per invocation. The one collision this admits is honest and benign: after a restart, a resumed run re-presents the same `callId`, and a fresh boundary can therefore mint a lease token identical to one issued pre-restart, so a very late result from the dead process could be accepted by the new one. That is not a safety defect — a colliding token implies the same `(capabilityId, callId)`, i.e. the same logical invocation, and side-effect-once for that invocation is guaranteed by the ADR-052 idempotency envelope, not by the lease token. Making the counter durable would add cross-process state to buy protection against a case that is already correct.
- **Consequences:**
  - Both obligations ADR-062 §6 deferred to issue #102 are discharged, and §6 is amended to say so rather than leaving a live forward reference in shipped code. `@tuvren/stream-ws` loses its internal sequencer and becomes carriage in the strict sense its own ADR claims for it — a breaking change to an `@experimental` `0.x` package, which is exactly the change ADR-062 anticipated.
  - **No new wire vocabulary.** Reattach uses the existing `WsHandshakeRequest.sessionId` + `cursor`; redelivery re-sends the existing `client_invocation` frame. That the design fits the shipped schemas unchanged is the strongest available evidence that the frame family ADR-060 settled was drawn at the right boundary. What is new is *semantics* — redelivery, grace window, dispatch timeout — which is why this ADR is accompanied by conformance checks rather than a schema revision.
  - Session **identity** (`sessionId` → live `RemoteClientSession`) remains host application state, not framework state: a registry belongs to the host that also owns authentication and multi-tenancy. The framework guarantees that a session *can* be reattached, never that some singleton knows where it lives.
  - **Trade-off recorded:** redelivery guarantees at-least-once presentation of an outstanding invocation and leans on `idempotencyKey` for at-most-once *effect*. A peer that ignores `idempotencyKey` and performs a non-idempotent side effect can therefore double it across a reconnect. The alternative — a delivery-acknowledgement sub-protocol — was rejected as re-solving at the session layer what the idempotency envelope already solves at the invocation layer, at the cost of new wire vocabulary and a second correlation handle for peers to get wrong.
  - **Deferred:** graduation of the `@tuvren/core/capabilities` chain out of `@experimental` per ADR-056. This ADR produces the network evidence that gating clause names, but graduation is an ADR-054 public-API stability commitment across several packages and is taken as its own reviewed change rather than as a tail-end commit of the change series that generated its evidence.
