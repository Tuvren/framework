### ADR-062 @tuvren/stream-ws: a Runtime-Agnostic WebSocket Binding for the Duplex Session Protocol

- **Status:** accepted.
- **Context:**
  - ADR-060 fixed the transport-agnostic duplex session frame vocabulary (`spec/host/session/`) and shipped `@tuvren/host-session`'s `createDuplexSessionBinding`, which stays deliberately at the in-memory frame-object level: wire encoding, handshake, close semantics, heartbeat, and backpressure were all reserved for the transport layer. Issue #100 names that layer: a remote-client host (browser extension ↔ sandboxed agent binary, FastBlocks first) needs a single bidirectional connection carrying the session protocol, and no WebSocket transport exists anywhere in production source.
  - ADR-061 added the resume vocabulary (`spec/streaming/resume/`): a wire-level sequencing envelope and opaque cursor whose semantics live in `@tuvren/stream-core`. Issue #100's transport must carry that cursor — in its handshake for reconnect requests and on its outbound event frames — without owning any cursor semantics.
  - The repository ships zero WebSocket runtime dependencies, and the runtime story diverges per environment: Bun has `Bun.serve` websockets and a client `WebSocket` global, browsers have the client global only, Node needs `ws` (or Node 22+'s client global). Binding the package to any one of those would narrow the surface below what its consumers (a browser content-script client and a Bun/Node server host) need.
- **Decision:**
  1. **New experimental package `typescript/streaming/ws` (`@tuvren/stream-ws`)**, sibling of `@tuvren/stream-sse`, with a matching spec sub-surface `spec/streaming/ws/` (packet `tuvren.framework.event-stream-ws`) following the established sub-surface nesting precedent. Dependencies: `@tuvren/stream-core` (sequencing/cursor carriage) and `@tuvren/host-session` (frame vocabulary and binding); peer `@tuvren/core`. No third-party runtime dependency.
  2. **Runtime-agnostic push-model socket seam.** The package never opens a socket. A host adapts its real socket to `WsSocketSink { send(data), close(code, reason?) }` and pushes every inbound message into `transport.ingest(data)` — the same push shape as `DuplexSessionBinding.dispatchInbound`, so the package works identically over `Bun.serve` websockets, the browser `WebSocket` global, Node `ws`, or an in-memory pair in tests. The alternative — shipping per-runtime adapters now — was rejected: adapters are trivial for hosts to write, and shipping them would force runtime-specific test matrices before any consumer exists (#102 is the demonstration home).
  3. **JSON text messages; one message per frame.** Every wire message is a JSON text frame discriminated by `kind`. Session frames ride an outer envelope — `{kind: "frame", cursor?, frame: SessionOutboundFrame}` outbound and `{kind: "frame", frame: SessionInboundFrame}` inbound — so the ADR-060 vocabulary is carried byte-for-byte as its own JSON object and the transport adds only carriage fields. `cursor` (the ADR-061 token) is attached only to `kind: "event"` session frames — the canonical event stream is the only sequenced, replayable surface; `client_invocation` and `session_rejection` frames are not replayable and carry no cursor. Binary WS framing is deliberately not used in v1: the canonical serialization (`serializeTuvrenStreamEvent`'s binary-payload marker) already round-trips binary content through JSON, and a native-binary-frame optimization would fork the serialization authority for one transport. Recorded as a revisitable trade-off.
  4. **In-band handshake, first message in each direction.** Client → server `{kind: "handshake", protocolVersion: "1", sessionId?, cursor?, authToken?}`; server → client `{kind: "handshake_ack", protocolVersion: "1", sessionId, resumeStatus: "resumed" | "out-of-window" | "unknown-turn" | "none"}`. `authToken` is an opaque host concern — the transport carries it to the host's accept callback and never inspects it. `resumeStatus` projects the ADR-061 `ReplayStatus` vocabulary plus `"none"` for a fresh connection presenting no cursor; on `"resumed"` the transport emits the replayed frames (from the host-supplied replay buffer) before live frames. `protocolVersion` remains a per-frame literal per ADR-060 — the handshake echoes it but does not negotiate.
  5. **Close-code vocabulary in the WS application range**, mirroring the session rejection-code discipline: `4000 handshake_invalid`, `4001 protocol_version_unsupported`, `4002 session_not_found`, `4003 auth_rejected`, `4004 heartbeat_timeout`, `4005 backpressure_exceeded`; `1000` for normal end-of-session close. Inbound frames that are schema-invalid or state-refused do NOT close the socket — they surface as ADR-060 `session_rejection` frames exactly as the binding defines; close codes are reserved for connection-level conditions the frame vocabulary cannot express.
  6. **Application-level heartbeat and bounded backpressure.** `{kind: "ping"}`/`{kind: "pong"}` messages (the abstract sink has no opcode surface) on a configurable `heartbeatIntervalMs`; any inbound message (not only `pong`) counts as liveness, since a peer actively sending frames is demonstrably alive — `pong` remains the answer a quiet peer produces because the transport always answers an inbound ping the same way. A timer that fires with no inbound activity since its ping was sent is half-open detection and closes `4004`. The outbound bound is a configurable byte budget (`maxBufferedBytes`) enforced against the sink's optional `bufferedAmount()` report (the WebSocket `bufferedAmount` concept); hosts whose sink cannot report buffering run without transport-level backpressure enforcement. When both the budget and the sink capability are present, overflow closes `4005` rather than silently dropping — a dropped frame would create a sequence gap the resume cursor could neither explain nor repair, while a close converts overflow into an honest reconnect-with-cursor.
  7. **Experimental posture** per ADR-056: `0.x` packet, `@experimental` exports, graduation gated on the #102 network evidence like the rest of the chain.
- **Consequences:**
  - The seam layering is now complete and strict: `@tuvren/core/capabilities` owns payload shapes, `spec/host/session/` owns frame semantics, `spec/streaming/resume/` owns cursor semantics, and `@tuvren/stream-ws` owns only carriage — handshake, heartbeat, close codes, and queueing. No frame or cursor semantic can drift into the transport without violating its packet's forbidden-authority rule.
  - A host integrates with three objects it already holds: an `ExecutionHandle`-backed `DuplexSessionBinding`, a replay buffer fed by one sequencer instance (ADR-061's wiring rule), and its own socket adapted to the two-method sink. #102 inherits a ready transport and owes only the network endpoint policy (timeouts, detach, lease durability) it was always scoped to own.
  - **Trade-off recorded:** JSON-only framing spends wire bytes on binary payloads (marker-encoded arrays) that native WS binary frames could carry raw; accepted for v1 to keep one serialization authority, revisitable behind a new envelope `kind` without breaking existing frames.
  - **Deferred:** issue #100's AG-UI co-tenancy note is deliberately out of scope for this ADR — AG-UI remains a separate read-only projection, and if it ever rides this socket it does so behind a new envelope `kind` per the extensibility path recorded above, keeping the control channel (session frames) and the rendering channel (AG-UI) distinguishable at the envelope level.
