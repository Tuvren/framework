### ADR-059 Advanced Composition Surface (`@tuvren/sdk/advanced`) and the `@tuvren/kernel-grpc-client` Leaf; Host Import Contract Extended (Amends ADR-057)

- **Status:** accepted. Amends ADR-057 §3 (host import contract) and §6 (Reference Host re-point). ADR-057 remains accepted for everything else it decided (the `createTuvren` home and instances-only signature, the `@tuvren/runtime` demotion, the amended ADR-054 stable-core enumeration). This ADR is a narrowing correction discovered while executing KRT-BJ002, not a reversal.
- **Context:**
  - ADR-057 §3 is normative: a host imports `@tuvren/core`, `@tuvren/sdk`, and its chosen leaf packages, and **never** imports `@tuvren/runtime` or the kernel packages (`@tuvren/kernel-protocol`, `@tuvren/kernel-runtime`) directly. §6 directs the 14 `@tuvren/runtime`-importing Reference Host files (`typescript/host/repl/src`) to re-point onto `createTuvren` plus curated primitives from `@tuvren/sdk`/`@tuvren/core` and leaf-package instances.
  - ADR-057's grounding audit (its Context item on the source `grep`) checked host-facing **types** only — it verified that `ApprovalRequest`/`DEFAULT_SCOPE`/`TuvrenRuntimeError` and the event/execution types already originate in `@tuvren/core`, and concluded the re-point is "a re-export-surface cleanup, not a type migration."
  - Executing KRT-BJ002 surfaced what that type-only audit missed: the Reference Host also consumes engine **factory functions** from `@tuvren/runtime` that are not on `@tuvren/sdk` or any `@tuvren/core` subpath and have no leaf-package equivalent:
    - `createOrchestrationRuntime` — builds **multi-agent** orchestration (`repl-scenarios.ts`, `repl-shell.ts`). `createTuvren` only ever builds a single default agent (`agents: { agent }`), so it cannot express the host's multi-agent scenarios.
    - `createRunnerRegistry` — builds multi-runner registries; `createTuvren` accepts a single `runner` factory and builds the registry internally.
    - `createTuvrenRuntime` — the primary **synchronous** `createReplHost` (used across `repl-scenarios.ts`, `repl-shell.ts`, tests, and the package index) constructs the runtime directly. `createTuvren` is asynchronous; consolidating onto it would turn a sync constructor async across ~16 call sites — a behavior change ADR-057 §6 did not intend.
    - `createGrpcRuntimeKernel` — the `rust-grpc` remote-kernel proving mode. It lived only in `@tuvren/runtime` (`src/lib/runtime-kernel-grpc.ts`), and no leaf gRPC-client package existed.
    - `createRuntimeKernel` — the local kernel constructor; it lives in `@tuvren/kernel-runtime`, which §3 also forbids the host from importing directly.
  - The Reference Host is the designated serious proving host (per `stack.md`), and these are exactly the capabilities it exists to prove: multi-agent orchestration, custom runner registries, and a remote Rust kernel over gRPC. `createTuvren` is single-agent/single-runner/local-kernel by construction and cannot substitute for them. So ADR-057's "re-export cleanup" is insufficient: without a curated home for these host-facing composition/kernel factories, the host must either reach into `@tuvren/runtime` (violating §3) or lose proving coverage.
- **Decision:**
  1. **Add a curated subpath `@tuvren/sdk/advanced`** exporting the low-level composition factories `createOrchestrationRuntime`, `createRunnerRegistry`, `createTuvrenRuntime`, and `createRuntimeKernel` (plus the option types a host needs to call them). This is the explicit "advanced composition" tier for hosts that build multi-agent orchestration, custom runner registries, or a bespoke local kernel beyond the batteries-included `createTuvren`. It re-exports from `@tuvren/runtime`/`@tuvren/kernel-runtime`, both of which `@tuvren/sdk` already lists as regular `dependencies` (ADR-057 item 1) — so install weight is unchanged. The root `@tuvren/sdk` surface (`createTuvren` + curated primitives) stays the default host entrypoint; `/advanced` is opt-in and separately importable so freeze tooling and documentation can treat it as an explicitly advanced surface.
  2. **Extract a new published leaf `@tuvren/kernel-grpc-client`** (`typescript/kernel/grpc-client/`) owning `createGrpcRuntimeKernel` and its `GrpcRuntimeKernelOptions`, together with the kernel-interop generated bindings and the gRPC codec relocated out of `@tuvren/runtime`. It is a host-facing kernel-transport leaf: a host that talks to a remote Rust kernel over gRPC installs it, constructs the `RuntimeKernel`, and passes it into `createTuvren`'s `kernel` option. `@tuvren/runtime` drops its `@connectrpc/connect`, `@connectrpc/connect-node`, and `@bufbuild/protobuf` dependencies and no longer re-exports the gRPC factory. The kernel-interop codegen output (`buf.gen.yaml` out-path, the `kernel-interop-governance` path constants, and the `kernel-interop-grpc` Nx project's `outputs`) moves from `typescript/runtime/src/lib/generated/kernel-interop` to the new leaf.
  3. **Extend ADR-057 §3's host import contract.** The host MAY additionally import `@tuvren/sdk/advanced` and the host-facing kernel-transport leaf `@tuvren/kernel-grpc-client`. It still MUST NOT import `@tuvren/runtime`, `@tuvren/kernel-protocol`, or `@tuvren/kernel-runtime` directly. The KRT-BJ003 automated host-boundary gate enforces exactly this partition: block `@tuvren/runtime`/`@tuvren/kernel-protocol`/`@tuvren/kernel-runtime`; allow `@tuvren/sdk` (root and `/advanced`) and `@tuvren/kernel-grpc-client`.
  4. **`@tuvren/kernel-grpc-client` joins the publishable leaf set** and `@tuvren/sdk/advanced` is part of the `@tuvren/sdk` stable-core surface (same package, an additional subpath). Both are in scope for KRT-BJ007 publication metadata, KRT-BJ008 release-version computation, and the ADR-054/056 freeze snapshot (the leaf as a host-facing package; `/advanced` as an additional entry of the already-frozen `@tuvren/sdk`).
- **Consequences:**
  - The Reference Host re-points with **no behavior change**: the synchronous `createReplHost` keeps its engine-path construction, now importing the factories from `@tuvren/sdk/advanced` and `@tuvren/kernel-grpc-client` instead of `@tuvren/runtime`. This satisfies KRT-BJ002's "typecheck and test lanes pass unchanged in behavior" acceptance criterion while honoring §3. (The pre-existing `createReplHostUsingCreateTuvren` path, broken by ADR-057's string-kind removal, is fixed to the instances-only shape in the same milestone.)
  - `@tuvren/runtime`'s dependency fan-in shrinks further (loses the three connect/protobuf transport deps), reinforcing the audit finding [E-03] intent that motivated ADR-057.
  - The freeze gate (ADR-054/056) snapshots `@tuvren/sdk` (root **and** `/advanced`), `@tuvren/core`, and the published leaf set including `@tuvren/kernel-grpc-client`.
  - **Trade-off recorded:** `@tuvren/sdk/advanced` deliberately exposes engine-level factories through the curated tier. It is an escape hatch for advanced hosts, kept off the default `@tuvren/sdk` root so the common surface stays minimal. It does **not** re-open the host→`@tuvren/runtime` coupling ADR-057 closed: the dependency edge is host→`@tuvren/sdk`→`@tuvren/runtime`, so `@tuvren/runtime` remains an internal package no host names directly, and the boundary gate still fails on any direct `@tuvren/runtime` import.
  - This ADR does not change any kernel/interop proto surface, the `createTuvren` signature, or the instances-only options shape; it only adds a curated subpath, a leaf, and two allowlist entries to the host contract.
