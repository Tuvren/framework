### ADR-065 The Idempotency Identity Is the Logical Call Identity, Not the Authority Epoch

- **Status:** accepted. Amends ADR-052 §1.
- **Context:**
  - ADR-052 §1 promises that "a re-dispatch after recovery presents the **same identity** for the same logical call," and derives that identity from `(runId, callId, fencingToken)`. The architecture layer repeats the promise as the reason the envelope exists at all: the dispatch envelope carries an idempotency identity "so a client environment can deduplicate a side effect it re-executes after a reconnect" (`.constitution/architecture/containers.md`, Client Endpoint Boundary).
  - **The implementation cannot satisfy that promise, because the third component is not stable.** `createFencingToken()` mints a *fresh* token on every lease renewal (`typescript/kernel/runtime/src/lib/runtime-kernel-runs.ts:542`), `ActiveRunLease.fencingToken` is documented as rotating on each renewal (`typescript/runtime/src/lib/runtime-core-liveness.ts:44-45`) and is overwritten on every renewal and piggybacked step result (`:206`), and that live value is what reaches `deriveIdempotencyKey` via `host.getActiveFencingToken(handle)` → `environment.fencingToken` (`typescript/runtime/src/lib/runtime-core-runner-support.ts:150`, `typescript/runtime/src/lib/tool-execution-helpers.ts:207-211`). The identity for one logical call therefore changes whenever the lease renews — not merely across a process restart, but *within a single long-running turn*, which is precisely the situation a long side-effecting call is most likely to be in.
  - The existing unit test certifies the defect rather than the contract: `typescript/runtime/test/idempotency-identity.test.ts` asserts that changing only the fencing token yields a different key. It tests that the function is a deterministic function of its arguments — true, and irrelevant — instead of testing ADR-052's actual claim, that the same logical call yields the same identity across a recovery.
  - The gap was found while reviewing ADR-063 (issue #102), whose reattach-and-redelivery design leans on the envelope for at-most-once *effect*. ADR-063's own analysis of lease-counter durability had adopted the same false premise, so this is not a theoretical inconsistency: the first design to depend on the guarantee immediately depended on it incorrectly.
- **Decision:**
  1. **The idempotency identity is derived from `(runId, callId)` alone.** `deriveIdempotencyKey(runId, callId)` drops the fencing token entirely. Both components are stable across recovery by construction: a resumed run keeps its `runId`, and recovery re-presents the same `callId` — a stability the framework already depends on elsewhere, since skip-completed-by-`callId` recovery (`docs/KrakenFrameworkSpecification.md` §4.9) would not work otherwise. The canonical injective JSON-array encoding and the sha256 hex digest are unchanged, so the value stays a fixed-length opaque token suitable for presenting to an external system.
  2. **The fencing token's role is unchanged and unshared: it is write authority, never identity.** ADR-052 §3 (client-result-as-proposal — a result becomes durable only through a commit under a valid run fencing token) is untouched and remains the mechanism that stops a stale owner mutating committed history. The defect was *conflation*: "which logical call is this?" and "may I commit this?" are orthogonal questions, and answering the first with the answer to the second made the first unstable for exactly as long as the second was doing its job.
  3. **The guarantee is stated in terms an external system can act on.** The envelope guarantees that every dispatch of one logical call — first attempt, framework retry, post-recovery re-dispatch, and ADR-063 redelivery after a reconnect — presents an identical key. It does not guarantee that a peer performs the effect once; that remains the peer's obligation, discharged by presenting the key to the external system. The framework's job is to make the key *worth* presenting.
  4. **Verification asserts the contract, not the function.** The existing "distinct triples produce distinct identities" fencing-token case is replaced by its inverse: the same `(runId, callId)` observed under *different* fencing tokens must yield the same key. A behavioral test drives a real recovery and asserts key stability across it, so the promise is proven at the level ADR-052 states it rather than at the level the helper happens to be implemented.
- **Consequences:**
  - ADR-052 §1's promise becomes true. ADR-063's redelivery design, the Client Endpoint Boundary's reconnect-dedup rationale, and any host that threads the key into an external call all rest on a guarantee that now holds.
  - **No durable-state or wire migration.** The key is never persisted: it is derived per dispatch and handed to `ToolExecutionContext.idempotencyKey` and `ClientInvocationEnvelope.idempotencyKey` (`spec/host/session/typespec/main.tsp`), both of which keep an unchanged `string` shape. Emitted *values* change, which is visible only to an external system mid-flight across the upgrade — accepted, since the alternative is leaving values that were never stable to begin with.
  - **Behavior change worth naming:** a framework retry of the same `callId` now presents the same key as the original attempt, where previously a renewal between attempts could produce a different one. That is the intended semantic — a retry of a logical call *should* dedup against the attempt that may already have succeeded externally — and is the whole reason a caller presents an idempotency key.
  - **Trade-off recorded:** `(runId, callId)` deliberately does not distinguish attempts, so a host that wants per-attempt external identity (for example to record each try separately) cannot recover it from this key. That is the correct default for a *dedup* key; per-attempt observability belongs to telemetry and the audit trail, which already carry attempt-level detail.
  - The optional-fencing-token branch and its absent-versus-empty-string normalization disappear with the parameter, simplifying the helper: a runtime with run-liveness leases and one without now derive identity identically, which is itself the honest outcome — liveness configuration never should have changed a call's identity.
