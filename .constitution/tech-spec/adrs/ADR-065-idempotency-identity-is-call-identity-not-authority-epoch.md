### ADR-065 The Idempotency Identity Is the Logical Call Identity, Not the Execution Attempt

- **Status:** accepted. Amends ADR-052 §1.
- **Context:**
  - ADR-052 §1 promises that "a re-dispatch after recovery presents the **same identity** for the same logical call," and derives that identity from `(runId, callId, fencingToken)`. The architecture layer repeats the promise as the reason the envelope exists: the dispatch envelope carries an idempotency identity "so a client environment can deduplicate a side effect it re-executes after a reconnect" (`.constitution/architecture/containers.md`, Client Endpoint Boundary).
  - **Two of the three components are not stable for one logical call, so the promise never held.**
    - The **fencing token** rotates on every lease renewal: `createFencingToken()` mints a fresh token on each renew (`typescript/kernel/runtime/src/lib/runtime-kernel-runs.ts:542`), `ActiveRunLease.fencingToken` is documented as rotating (`typescript/runtime/src/lib/runtime-core-liveness.ts:44-45`) and is overwritten on every renewal and piggybacked step result (`:206`).
    - The **`runId`** is less stable still, and this is the more damaging half. A Run is one execution *attempt*, not the interaction: the kernel spec states that "multiple Runs may serve the same Turn (pause/resume creates new Run from pause point)" (§5.2) and that a replacement execution after preemption "MUST be a new Run. Reopening the stale Run is illegal" (§5.2). The framework mints one per **ReAct iteration** (`typescript/runtime/src/lib/runtime-core-iteration.ts:579`, stamped onto the batch environment at `:718`), per approval resume (`typescript/runtime/src/lib/runtime-core-tool-resume.ts:176`), and at each recovery. The identity therefore churned in a perfectly healthy loop, not merely across the failures the envelope exists to survive.
  - The result was an idempotency key that changed under exactly the conditions it was supposed to be invariant to, which makes it worse than no key: a peer or tool that trusted it for external dedup would fail to dedup precisely on retry, resume, and recovery.
  - **Existing verification certified the defect rather than the contract.** `typescript/runtime/test/idempotency-identity.test.ts` asserts that changing only the fencing token yields a *different* key — a true statement about the function and an irrelevant one about the promise — and its own comment concedes that "in real preemption recovery the runtime mints a fresh runId ... so the live triple is not stable across recovery," deferring reconciliation elsewhere. No test anywhere drives a real pause/resume or recovery and asserts the key is unchanged across it.
  - The gap surfaced while reviewing ADR-063 (issue #102), whose reattach-and-redelivery design leans on the envelope for at-most-once *effect*. The first design to depend on the guarantee immediately depended on it incorrectly, which is the strongest possible argument that the guarantee needed to become real rather than be worked around.
- **Decision:**
  1. **The idempotency identity is derived from `(turnId, callId)`** — the logical call identity. `deriveIdempotencyKey(turnId, callId)` replaces the `(runId, callId, fencingToken)` triple. The canonical injective JSON-array encoding and the sha256 hex digest are unchanged, so the value remains a fixed-length opaque token suitable for presenting to an external system.
  2. **This is the minimal tuple that is both stable and unique.** `turnId` identifies the user-visible interaction unit, which is precisely the scope designed to span many Runs, and every resume and recovery path threads the existing `handle.turnId` through unchanged (`runtime-core-tool-resume.ts:181`, `:532`; recovery returns the pre-existing `turnId` read back from the kernel's expired-run record, `runtime-core-expired-recovery.ts:154`). `callId` is framework-generated exactly once when the assistant's tool-call part is first parsed (`typescript/runners/react/src/lib/react-runner-stream-support.ts:493`) and never re-minted; recovery re-presents it from committed or staged state, a stability that skip-completed-by-`callId` recovery already depends on (framework spec §4.9, §8.6). `(threadId, callId)` would also be stable but is a looser scope than necessary, since a thread spans many turns and forked branches.
  3. **The fencing token's role is unchanged and unshared: it is write authority, never identity.** ADR-052 §3 (client-result-as-proposal — a result becomes durable only through a commit under a valid run fencing token) is untouched and remains the mechanism that stops a stale owner mutating committed history. The defect was *conflation*: "which logical call is this?" and "may I commit this?" are orthogonal questions, and answering the first with the answer to the second made the first unstable for exactly as long as the second was doing its job. Keying on `runId` compounded it by answering the first question with "which attempt is this?" instead.
  4. **The guarantee is stated in terms an external system can act on.** The envelope guarantees that every dispatch of one logical call — first attempt, framework retry, post-approval-resume re-dispatch, post-recovery re-dispatch, and ADR-063 redelivery after a reconnect — presents an identical key. It does not guarantee that a peer performs the effect once; that remains the peer's obligation, discharged by presenting the key to the external system. The framework's job is to make the key *worth* presenting.
  5. **Verification asserts the contract, not the function.** The fencing-token case in "distinct triples produce distinct identities" is replaced by its inverse — the same `(turnId, callId)` observed under different run and fencing identities must yield the same key — and a behavioral test drives a real approval pause and resume, asserting that the `idempotencyKey` handed to the tool before the pause is identical to the one handed to it after, across the new Run the resume creates. That closes the coverage hole the old test's own comment documented.
- **Consequences:**
  - ADR-052 §1's promise becomes true for the first time. ADR-063's redelivery design, the Client Endpoint Boundary's reconnect-dedup rationale, and any host or tool that threads the key into an external call now rest on a guarantee that holds.
  - **No durable-state or wire migration.** The key is never persisted: it is derived per dispatch and handed to `ToolExecutionContext.idempotencyKey` and `ClientInvocationEnvelope.idempotencyKey` (`spec/host/session/typespec/main.tsp`), both of which keep an unchanged `string` shape. Emitted *values* change, which is visible only to an external system mid-flight across the upgrade — accepted, since the values being replaced were never stable enough to be relied on.
  - **Behavior change worth naming:** a framework retry of the same `callId`, and every iteration of a loop that re-dispatches it, now present the same key as the original attempt, where previously each attempt could differ. That is the intended semantic — a retry of a logical call *should* dedup against the attempt that may already have succeeded externally — and is the whole reason a caller presents an idempotency key.
  - **Trade-off recorded:** `(turnId, callId)` deliberately does not distinguish attempts, so a host that wants per-attempt external identity (for example to record each try separately) cannot recover it from this key. That is the correct default for a *dedup* key; per-attempt observability belongs to telemetry and the `tool.audit` trail, which already carry attempt-level detail.
  - The optional-fencing-token branch and its absent-versus-empty-string normalization disappear with the parameter. A runtime with run-liveness leases and one without now derive identity identically, which is itself the honest outcome — liveness configuration never should have changed a call's identity.
  - `docs/KrakenFrameworkSpecification.md` §4.8's side-effect-once rule is amended in the same change, since it also stated the `(runId, callId, fencingToken)` triple.
