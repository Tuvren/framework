### ADR-049 Isolation-by-Construction: Durable Identity Is Scope-Resolved

> **Path/terminology note (2026-07, epic #87, M10.3b):** This ADR is a historical decision record; its body below is preserved verbatim and is not rewritten. Epic #87 renamed "driver" → "runner" repo-wide (e.g. the ReAct Driver → the ReAct Runner, `@tuvren/driver-api` → the `@tuvren/core/runner` subpath, `driverId`/`DriverKind` → `runnerId`/`RunnerKind`) and relocated `boundaries/<area>/...` paths into `spec/<port>/...` (language-neutral authority) plus `typescript/<area>/...` / `rust/<area>/...` (language-specific implementations); `implementations/<lang>/` subtrees moved to top-level `typescript/`/`rust/` trees. Any `driver` term or `boundaries/`-rooted path below reflects the pre-epic-#87 name or location as of this decision's date; see `.constitution/tech-spec/guidelines.md` and `.constitution/tech-spec/stack.md` for the current map. This note does not change the decision recorded below.

- **Status:** accepted
- **Context:** PRD v0.10.0 CAP-P0-065 requires isolation-by-construction, and Architecture v0.10.0 names the cross-scope leak failure class: a globally content-addressed object store turns `store.has(hash)` into a cross-tenant existence oracle, and an unscoped enumeration leaks one tenant's threads to another. Today the TypeScript backends key objects purely by content hash (`objects.hash` PRIMARY KEY in `boundaries/kernel/implementations/typescript/backend-sqlite/migrations/0001_initial_schema.sql`), which is a single global identity space within a backend. With ADR-048 binding scope at construction, the remaining decision is whether content identity is resolved globally or within a scope.
- **Decision:** Durable identity is resolved **within a scope**. Content addressing is `(scope, hash)`, not `hash` alone:
  1. Object, TurnTree, TurnNode, schema, staging, and enumeration reads/writes are all confined to the constructing scope. Identical content in two scopes is stored independently; there is no cross-scope deduplication.
  2. `store.has`, `store.get`, `branch.list`, `thread.list`, and every Durable-Read Surface operation can only observe content in their own scope. An existence check, enumeration, or read can never reveal that another scope holds given content.
  3. Realization is per backend and is a substrate detail, not a kernel-semantic change: a scope-per-file or scope-discriminator column for SQLite, a row-level-isolated connection/schema for PostgreSQL, and a scope-keyed map for memory. The kernel's hashing algorithm and canonical-blob identity are unchanged; only the substrate's lookup is scoped.
- **Consequences:** Cross-tenant dedup is deliberately surrendered to gain isolation-by-construction; this is recorded in PRD as a scope distinction (scoped vs. global dedup), not a regression. Per-scope identity makes reclamation (ADR-051) and erasure naturally per-scope, so tenant offboarding is dropping a scope plus destroying its keys. `docs/KrakenKernelSpecification.md` §2.3 is amended to state that hash identity is resolved within the scope the durable substrate is constructed against (the kernel still computes the hash; the substrate scopes the lookup). Conformance for the SaaS-readiness epics asserts that no read, enumeration, or existence check crosses a scope across all three backends.
