### ADR-060 Duplex Session Protocol: a Session-Frame Vocabulary Above the Canonical Stream

- **Status:** accepted.
- **Context:**
  - The canonical event stream (§4.5) is one-directional: `TuvrenStreamEvent` flows runtime-to-host, and every client-to-agent interaction — `resolveApproval`, `steer`, `cancel`, and a Tuvren-client capability's client-reported result — is an in-process call on the `ExecutionHandle` (`typescript/core/src/lib/runtime-contract-shapes.ts:1821-1828`) or the `AttachedClientEndpoint` (`typescript/core/src/lib/capability-shapes.ts:352-388`). That is sufficient for a host process that embeds Tuvren directly, but it has no answer for a **remote** session peer — a browser tab, a mobile client, or any process that is not the host process itself and can only reach the runtime over a wire.
  - Issue #99 names the forcing function: FastBlocks is the first consumer that needs a duplex session over a network transport, not an in-process handle. Without a framework-owned vocabulary, FastBlocks (and every subsequent host with the same need) would invent its own ad hoc protocol for wrapping outbound events and submitting inbound control frames — exactly the kind of per-host protocol fragmentation the Host Integration Boundary exists to prevent.
  - Five companion issues are already named and deliberately out of scope here: WebSocket transport (#100), a resumable cursor over reconnects (#101), the network-facing endpoint/timeout/disconnect policy (#102), AG-UI attribution (#103), and the host-supplied result-sanitization hook (#104). This ADR settles only the transport-agnostic frame vocabulary those efforts sit on top of.
- **Decision:**
  1. **New experimental spec sub-surface `spec/host/session/`**, nested under the existing host port the same way `spec/tools/mcp/` nests under `spec/tools/` and `spec/streaming/sse/` nests under `spec/streaming/` — a focused sub-authority for one protocol concern rather than a new top-level port. It gets its own authority packet, `tuvren.framework.host-session`, with TypeSpec source and generated JSON Schema artifacts as authority, following the same TypeSpec-to-JSON-Schema pipeline every other authority packet in `spec/` uses. It lands in a later milestone; this ADR fixes the shape the packet will carry.
  2. **Session-frame-only promotion.** The canonical `TuvrenStreamEvent` union (26 variants, §4.5) is not extended with a 27th session-control variant. Instead, the session protocol's outbound side wraps a canonical event inside a session frame (`{kind: "event", event: TuvrenStreamEvent}`) and adds session-owned frame kinds alongside it. The alternative considered — adding session-control variants directly to `TuvrenStreamEvent` — was rejected: the canonical stream is consumed by every host, including in-process hosts that never see a remote peer, and its `EventSource`-keyed, provider/runner-attributed vocabulary has no natural home for a per-dispatch `leaseToken`. Folding leaseToken-bearing frames into the canonical union would mean either broadcasting the lease token on the same channel every in-process consumer reads (leaking session-channel material into the canonical stream) or defining canonical variants with fields most consumers can never populate. Confining the lease token to the session channel between the runtime and the executing client endpoint is the whole rationale for keeping the two vocabularies separate: session-frame-only promotion is a trust-boundary decision, not just a namespacing one.
  3. **The frame family and rejection semantics.** Frames share `protocolVersion: "1"` (a per-frame literal, not a session-wide negotiated value), `sessionId` (stable across a `resolveApproval` handle replacement — the identity a remote client tracks across the runtime's internal handle churn), and a `kind` discriminator. Outbound: `{kind: "event", event}` | `{kind: "client_invocation", invocation: ClientInvocationEnvelope}` | `{kind: "session_rejection", rejection: {correlationId, code, message, details?}}`. Inbound: `{kind: "client_result", correlationId, result: ClientReportedResult}` | `{kind: "approval_response", correlationId, response: ApprovalResponse}` | `{kind: "steer", correlationId, signal: InputSignal}` | `{kind: "cancel", correlationId}`. `correlationId` is client-generated and echoed back on rejection so a client can always match a rejection to the frame it sent. No inbound frame is ever silently dropped: a schema-invalid frame produces `session_frame_invalid`; a schema-valid frame the held handle refuses because the session is in the wrong state (an `approval_response` while not paused, a `cancel` racing an already-applied approval) produces `session_frame_wrong_state`, carrying the underlying `TuvrenRuntimeError` code in `details.runtimeErrorCode`; and a `client_result` whose `callId` matches no pending dispatch produces `capability_result_stale` (`typescript/core/src/lib/capability-error-codes.ts:59`), reusing the existing capability error code rather than minting a session-local synonym. `session_frame_invalid` and `session_frame_wrong_state` are session-surface-owned codes because they describe wire/session-state conditions the capability vocabulary has no concept of; `capability_result_stale` is reused because it already means exactly this at the capability layer. This produces two composed staleness layers by design: the session layer rejects a `client_result` with no matching pending dispatch, and the inner per-dispatch `leaseToken`/`callId` echo check already enforced at the Client Endpoint Boundary (`typescript/runtime/src/lib/client-endpoint-boundary.ts`, KRT-AZ003) still runs independently underneath it. Neither layer substitutes for the other.
  4. **Reference binding `@tuvren/host-session`** (`typescript/host/session/`, later milestone): `createDuplexSessionBinding(handle) → {sessionId, outbound(), dispatchInbound(frame), clientEndpoint: AttachedClientEndpoint}`. The binding holds the current `ExecutionHandle`, tracks the replacement handle `resolveApproval` returns, and re-bridges the resumed handle's `events()` into one continuous `outbound()` stream so a remote peer never sees the internal handle-replacement seam as a stream discontinuity. It ships a minimal in-process frame-sourcing `AttachedClientEndpoint`: `dispatch(envelope)` emits a `client_invocation` frame and resolves on the correlated inbound `client_result` via a pending-promise map keyed by `callId`. Network policy — timeouts, disconnect handling, retry — is explicitly deferred to #102; this binding only defines the frame-level contract and the handle-replacement re-bridge duty, not a network endpoint implementation.
  5. **Experimental posture.** The whole surface — the packet at `0.x`, the promoted `ClientInvocationEnvelope`/`ClientReportedResult` schemas, and the TS reference binding — stays `@experimental` per ADR-056's TSDoc-tag convention. Graduation is deferred until real network evidence exists; issue #102 (the network-facing endpoint and its timeout/disconnect/repl policy) is the named forcing function for that evidence, not a date.
- **Consequences:**
  - The remote-client trust boundary is now explicit at the protocol level rather than implied: every inbound frame is schema-validated before it can touch a handle, and a client-reported result remains a proposal committed only under valid run authority (the existing ADR-052 rule), unchanged by this ADR.
  - The canonical stream is untouched — no existing consumer, adapter, or conformance plan that reads `TuvrenStreamEvent` needs to migrate, because the 26-variant union gained nothing and lost nothing.
  - This ADR is a layering foundation for a named sequence of companion issues: #100 (WebSocket transport), #101 (resume cursor), #102 (network-facing endpoint and repl/timeout policy), #103 (AG-UI attribution), and #104 (sanitization) all build on this frame vocabulary without changing it. AG-UI stays a strictly read-only projection (`stack.md` §1.1); the control channel this ADR defines never tunnels through AG-UI `CUSTOM` frames, so #103's attribution work has a stable vocabulary to project from rather than a moving target.
  - **Trade-off recorded:** `sessionId` and `correlationId` add wire overhead that has no equivalent in the in-process `ExecutionHandle` path — a host that never leaves one process pays nothing, because it simply never imports `@tuvren/host-session` and never constructs a frame. The overhead is confined to hosts that opt into the remote-session surface.
