### ADR-051 Data Lifecycle: Reachability Reclamation Primitive Plus Crypto-Shredding Erasure

- **Status:** accepted
- **Context:** PRD v0.10.0 CAP-P0-066/067 promote garbage collection from out-of-scope to an in-scope data-lifecycle capability, reframed as a compliance primitive (retention limits, right-to-erasure, tenant offboarding). Architecture v0.10.0 adds the Data Lifecycle, Reclamation & Erasure Model. Two facts make this tractable now: reachability in the content-addressed DAG is fully structural (the kernel can compute it without application semantics), and isolation-by-construction (ADR-049) makes the work per-scope. The hard tension is that immutable Merkle lineage cannot be edited to remove a subject's data without breaking every downstream hash.
- **Decision:** Split the problem into a kernel-owned mechanism and a host-owned policy, and reconcile erasure with immutability via crypto-shredding:
  1. **Reclamation (kernel mechanism).** Add a capability-gated maintenance primitive that marks durable state reachable from live roots — non-archived branch heads, thread roots, and active-run staged work — and sweeps only the unreachable remainder within the scope. It is grace-windowed: it never releases state newer than the oldest active execution lease / in-flight write horizon, so it cannot race recovery. Cross-branch structural sharing is handled naturally because a shared object stays marked via any live root. The `BackendCapability` descriptor advertises reclamation support; object-store-style backends may reclaim differently or advertise non-support. The kernel decides reachability (structural); the host decides retention (which threads/scopes are still wanted) and calls reclamation through the framework maintenance surface. This is specified in `docs/KrakenKernelSpecification.md` (new reclamation section, §9 capability); the deferred-GC note is removed.
  2. **Erasure (crypto-shredding, host-keyed).** Sensitive untrusted-edge payloads — provider, client, MCP, and tool results, plus carried provider continuity artifacts — are encrypted with a host-held key before `store.put`, so the kernel stores opaque ciphertext blobs (it remains oblivious; ADR-001 preserved). "Erase a subject" = the host destroys the relevant key, rendering the payloads unrecoverable while the lineage hash structure and the audit shape of what remains stay intact. Keys are host-held; Tuvren never takes on key management and stores only ciphertext plus a key-reference. The encrypt/decrypt seam lives at the framework edges that produce these payloads (Provider Gateway, Tool Execution Gateway, MCP Client Container, Client Endpoint Boundary), consistent with the Secret Isolation Model.
  3. **Composition with tenancy.** Under ADR-049 both reclamation and erasure are per-scope, so tenant offboarding is dropping the scope partition plus destroying its keys (Architecture flow §4.17).
- **Consequences:** A content-addressed agent runtime becomes legally deployable as a SaaS substrate: retention limits and right-to-erasure are satisfiable without rewriting committed history. The kernel gains exactly one new mechanism (reclamation) and no new record semantics; shredding is entirely a substrate/edge concern (ciphertext blobs + host keys). Reclamation must be conformance-proven not to release reachable state and not to race active leases; erasure must be proven to leave lineage structure intact while making the payload unrecoverable. Branch-merge GC and cross-scope dedup remain out of scope. Decryption-on-read means a shredded payload surfaces as a typed unavailable/erased result rather than a crash.
