### ADR-058 Construction-Time Funnel Routing: Telemetry Destination Contract With One-Directional Failure Isolation

- **Status:** accepted
- **Context:** PRD v0.12.0 CAP-P0-071 establishes that one runtime execution emits two distinct data funnels born together: the content funnel (session-critical durable state) and the telemetry funnel (operational metadata that no session depends on for correctness). CAP-P0-072 requires that the host decide, at construction time, how each funnel is persisted — split to separate destinations, unified into one store, or mixed across substrates — without changing session behavior. CAP-P1-073 asks for ready-made durable telemetry persistence but marks it additive and deferrable behind the stable contract. Architecture v0.12.0's Data Funnel Separation Model (`.constitution/architecture/resilience.md` §5) states the invariant in normative terms: the failure-isolation direction is one-directional — session correctness, continuity, and recovery never depend on the telemetry funnel, so a telemetry-destination failure must degrade telemetry only (dropped or bounded-buffered records, surfaced as an operational signal) and must never fail, block, or corrupt a content-funnel commit; the Kernel syscall surface stays funnel-unaware, mirroring the scope seam. Flow §4.19 (`flow-two-funnel-routing.md`) traces the split-topology happy path and the load-bearing unhappy path (destination unreachable mid-turn) and states the topology-equivalence claim: split, unified, and mixed-substrate routing change only the SDK construction-time seam, never the arrows below it. The physical surface today only realizes half of this: `TuvrenTelemetrySink` (`typescript/core/src/telemetry/index.ts`, ADR-042) is a push-based, fire-and-forget consumer of `TelemetrySpan`/`TelemetryEvent`, wired through `createTuvren({ telemetry })` and emitted by `typescript/runtime/src/lib/runtime-core-telemetry.ts`. It has no delivery contract, no backpressure/buffering semantics, and no operational-signal channel for delivery failure — it is an emission interface, not a destination contract, and nothing in the codebase names a durable telemetry destination. The content funnel's destination is already first-class: the `backend` option on `CreateTuvrenOptions` is the Durable State Boundary. Without a symmetric telemetry-destination contract, CAP-P0-072's routing decision has no seam to bind to, and CAP-P1-073's deferred ready-made persistence packages have no stable contract to implement against before the SDK freeze (ADR-054).
- **Decision:** Introduce the telemetry-destination contract and its construction-time routing seam now, deferring only concrete destination adapters, in six parts:
  1. **`TelemetryDestination` contract in `@tuvren/core/telemetry`.** A durable delivery target for telemetry-funnel records, distinct from `TuvrenTelemetrySink`: `deliver(batch: ReadonlyArray<TelemetrySpan | TelemetryEvent>): void` accepting correlated records in the existing canonical vocabulary and lineage keys (unchanged). Buffering and backpressure are owned by the destination adapter and declared on its descriptor (bounded buffer size, drop-oldest-on-overflow policy, or an equivalent stated policy) — the contract does not mandate one universal buffering strategy because destinations differ (in-process capture, network-bound store, batching exporter). Delivery failures are never thrown into the caller; they are surfaced through an operational-signal channel (a host-observable callback carried on the destination, analogous to telemetry-about-telemetry) so an operator can alert on funnel health without the runtime depending on that signal for correctness.
  2. **Generalized `createTuvren({ telemetry })` seam.** The `telemetry` option on `CreateTuvrenOptions` (ADR-040, home moved to `@tuvren/sdk` per ADR-057) widens from `TuvrenTelemetrySink` to accept a `TuvrenTelemetrySink`, a `TelemetryDestination`, or a route object `{ sink?: TuvrenTelemetrySink; destination?: TelemetryDestination }` combining both. This union *is* the funnel-routing seam required by CAP-P0-072: split topology is `createTuvren({ backend: tenantStoreBackend, telemetry: { destination: centralizedDestination } })`; unified topology is a host-authored `TelemetryDestination` backed by the same store as `backend`; mixed-substrate topology is any other pairing. As with the Scope seam (ADR-048), routing is bound once at construction; no runtime API mutates it afterward, keeping funnel topology out of the hot path and out of kernel syscalls.
  3. **One-directional failure-isolation invariant (normative).** Every telemetry emission point in the runtime (`runtime-core-telemetry.ts` and equivalents) remains fire-and-forget with respect to session execution, extended from sink to destination: an error raised by a `TelemetryDestination.deliver` call, or by a `TuvrenTelemetrySink`, is caught at the telemetry boundary, converted to the operational signal from part 1, and can never fail, block, or delay a kernel checkpoint or a content-funnel commit. Session recovery (ADR-045) never reads telemetry as an input to its decisions — recovery correctness is derived from durable lineage alone.
  4. **Cross-funnel hygiene.** No content payload (messages, tool results, durable state) is placed on the telemetry funnel absent the host's explicit routing decision (e.g., deliberately mirroring a payload into a destination it also owns). ADR-044's secret-screening obligations apply to the telemetry funnel unchanged and are not weakened by the destination contract. The audit finding that provider-bridge `requestBody` and response headers bypass ADR-044 screening on their way into durable state and the canonical event stream (`.constitution/reports/audit-2026-07-04-170703-post-epic-87-baseline.md`, finding [B-01], evidence at `typescript/providers/bridge-ai-sdk/src/lib/ai-sdk-provider-bridge-utils.ts:437`) is cited here as motivating evidence that shared screening boundaries must not silently narrow when a new outbound surface is added; remediating that specific gap is execution-plan scope, not part of this ADR.
  5. **Conformance.** New checks in the telemetry conformance plan assert: (a) identical session results for the same turn with the telemetry destination healthy versus unavailable (the flow §4.19 unhappy path); (b) no content payload appears on the telemetry funnel under default routing; (c) a destination delivery failure surfaces an operational signal rather than an exception or a silently dropped-without-signal record. The kernel syscall surface and the gRPC interop layer stay funnel-unaware — no proto change is introduced by this contract, matching the Data Funnel Separation Model's statement that the Kernel Boundary stays scope-free and funnel-unaware.
  6. **Deferred destination adapters.** Concrete, ready-made destination adapter packages (e.g., store-backed telemetry destinations satisfying CAP-P1-073) are deferred behind this stable contract. They ship additively, post-freeze, named in a future epic. The contract ships now specifically so the SDK freeze (ADR-054) already accounts for the funnel-routing surface, avoiding the freeze-before-scope failure class where a load-bearing seam is discovered only after the public API is frozen.
- **Consequences:** `TuvrenTelemetrySink` remains a fully valid, unchanged construction argument — the `telemetry` option's widening to a union type is backward-compatible and lands pre-freeze. The telemetry boundary in `@tuvren/runtime` gains a small, well-scoped buffering/operational-signal responsibility it did not have before. Hosts gain split, unified, and mixed-substrate topologies with zero session-behavior difference between them, satisfying CAP-P0-072 and flow §4.19's topology-equivalence claim. Conformance plans grow a funnel-isolation check set (healthy-vs-unavailable destination equivalence, no-content-on-telemetry-by-default, failure-to-signal mapping). Future destination adapter packages are semver-additive against the now-stable `TelemetryDestination` contract, so CAP-P1-073 can be delivered later without another breaking seam change. The ADR-044 secret-screening gap named in the post-epic-87 audit — open when this ADR was accepted — was closed by Epic BK's `KRT-BK004`: provider-bridge `bridgeExtras` request-body and response-header values are now screened for secret-shaped content (`screenValueForSecretPatterns` in `@tuvren/core/security`) before reaching durable records or the canonical event stream, proven by a new pattern-shaped `secretPatternAbsence` conformance assertion. This ADR did not close it itself, only named it so the telemetry-destination surface was not treated as already remediated at acceptance time.
