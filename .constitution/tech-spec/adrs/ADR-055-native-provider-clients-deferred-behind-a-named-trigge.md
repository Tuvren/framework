### ADR-055 Native Provider Clients Are Deferred Behind a Named Trigger; the Bridge Gets a Fidelity Audit

- **Status:** accepted
- **Context:** PRD v0.10.0 keeps native first-party provider clients out of the current line and records them as a named deferral. ADR-005 already keeps the AI SDK bridge as the baseline and lists `@tuvren/provider-openai`/`-anthropic`/`-google` as future additive packages. AI SDK v6 (`ai@6.0.142`) models tools as `function` vs provider-executed (`providerExecuted: true` + `dynamic: true`), which Tuvren's richer four-execution-class model already exceeds; but AI SDK has a known open issue where `parseToolCall` mis-validates provider-executed tools against the user's tool map and injects spurious errors (vercel/ai #10888), and the stateful-provider frontier (OpenAI Responses, Anthropic `cacheControl`) is round-tripped only as opaque `providerMetadata`. The strategic question is when native clients become worth their maintenance and dependency cost.
- **Decision:**
  1. **Deferral with a named trigger.** Native Anthropic/OpenAI/Gemini clients are not built in the current line. They are triggered when either condition holds: (a) provider statefulness cannot be faithfully round-tripped through the baseline `LanguageModelV3` contract as reconstructable continuity artifacts (ADR-053), or (b) Tuvren's context-manifest knowledge could materially cut token cost or latency (e.g. precise Anthropic `cacheControl` breakpoint placement) but the bridge cannot express it. When triggered, native clients are additive packages implementing the existing Tuvren-owned provider contract (`boundaries/providers/contracts/provider-api`), never a replacement of the bridge or the contract.
  2. **Bridge fidelity audit (now).** Before any native work, audit the baseline bridge's `providerExecuted`/`dynamic` round-trip fidelity against the parseToolCall landmine: confirm provider-executed tool calls/results are attributed to the provider-native execution class without spurious validation errors, that per-class observation limits hold, and that continuity artifacts are extracted into lineage and re-injected (the AY005 multi-turn `providerContinuity` round-trip gap noted in the TechSpec changelog is the concrete follow-up). This audit is a SaaS-readiness epic deliverable, not a native-client deliverable.
- **Consequences:** The runtime keeps one provider integration to maintain while it stabilizes and ships, consistent with the operator's "native is desirable but not the key path until stable" direction. Because the provider contract is already the seam (ADR-005), native clients remain a low-architectural-risk additive step whenever the trigger fires. The accepted statelessness token cost (ADR-053) is the standing business case that will eventually fire trigger (b). The audit hardens the one integration the runtime actually ships on now.
