### ADR-048 The Tenancy-Agnostic Scope Seam Is Bound at Backend/Connection Construction

- **Status:** accepted
- **Context:** PRD v0.10.0 CAP-P0-064 makes a tenancy-agnostic scope seam in-scope, and Architecture v0.10.0 adds the Tenancy & Scope Isolation Model: the runtime exposes a Scope (a host-bound partition identity) but never defines, authenticates, routes, or discovers tenants. The host must be free to build any topology — shared store with row-level isolation, store-per-tenant, central-metadata-plus-per-tenant-store, or fully decentralized — on one SDK. A competing design would thread a `scope`/`tenant` argument through the kernel syscall surface so the kernel enforces isolation; that was evaluated and rejected (resolved fork Q1) because it would churn the kernel protocol, the gRPC interop (`boundaries/kernel/interop/grpc/proto/**`), all three backends, the `RuntimeKernel` interface, and the about-to-freeze public surface, and it would push tenant-shaped policy into a deliberately data-only/protocol-only kernel (ADR-001).
- **Decision:** Scope is bound when a backend (or its connection/session) is constructed, not passed per syscall. Concretely:
  1. Backend factories (`@tuvren/backend-memory`, `@tuvren/backend-sqlite`, `@tuvren/backend-postgres`) accept a host-supplied scope binding at construction. `@tuvren/backend-memory` keys its in-memory stores by scope; `@tuvren/backend-sqlite` realizes a scope as a database file (or an explicit scope column for a shared file); `@tuvren/backend-postgres` realizes a scope through a row-level-isolated, host-supplied connection/pool (e.g. a session that sets the tenant discriminator the host's row-level-security policy reads) or a dedicated schema/database. The host owns the mapping; the backend realizes the isolation.
  2. The kernel syscall surface stays **scope-free**: no new syscall, no new syscall argument, no gRPC proto change. The kernel computes durable identity over the canonical blob exactly as today; the durable substrate the kernel writes to is already scoped by construction.
  3. The host-facing Durable-Read Surface (`TuvrenRuntime.listThreads` et al.) needs no scope parameter, because the runtime instance is constructed against one scope-bound backend. A host serving many tenants constructs one scoped runtime per tenant request (cheap for embedded backends; a per-request scoped connection for shared backends).
  4. `createTuvren(...)` and `createRuntimeKernel(...)` thread the host's backend choice through unchanged; the scope binding lives entirely in the backend construction the host supplies.
- **Consequences:** Tenancy is added without altering the kernel protocol, its cross-language interop, or the public host-facing read signatures, which protects the SDK freeze (ADR-054). Isolation correctness depends on the backend honoring the scope binding (realized and conformance-checked per backend in the SaaS-readiness epics), not on the kernel. Cross-tenant discovery, access-control policy, and tenant routing remain explicitly host-owned (PRD out-of-scope). The kernel remains data-only/protocol-only (ADR-001). This ADR is the precondition for ADR-049 (isolation-by-construction), ADR-051 (per-scope reclamation/erasure), and the freeze sequencing in ADR-054.
