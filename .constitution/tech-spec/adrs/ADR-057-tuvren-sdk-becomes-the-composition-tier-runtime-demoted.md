### ADR-057 `@tuvren/sdk` Becomes the Slim Convenience/Composition Tier; `@tuvren/runtime` Is Demoted to Internal Engine (Amends ADR-037/040/054)

- **Status:** accepted. Amends ADR-037's "`@tuvren/runtime` becomes the slim convenience package with one root export exposing `createTuvren({...})`", ADR-040's "Add `createTuvren({...})` as the sole root export of `@tuvren/runtime`", and ADR-054's stable-core enumeration ("`@tuvren/core` primitives ... the `ExecutionHandle`/`awaitResult` surface, `createTuvren`, and the published leaf packages"). Those ADRs remain accepted for everything else they decided: `@tuvren/core`'s subpath export map (ADR-037), the `CreateTuvrenOptions` lifecycle including `[Symbol.asyncDispose]` (ADR-040), and the freeze sequencing after tenancy/data-lifecycle work (ADR-054).
- **Context:**
  - PRD v0.12.0's Distribution & API Stability constraint hardens the host-facing boundary: the curated SDK boundary is the only supported host-facing surface, and internal runtime machinery below it is not a host-facing API. Architecture v0.12.0 adds a host-facing-tier principle and records a risk that the Reference Host (`typescript/host/repl`) imports `@tuvren/runtime` directly rather than going through a curated composition surface.
  - That risk is confirmed in the current tree: `typescript/host/repl/package.json` depends on `@tuvren/runtime` (`workspace:*`) and does **not** depend on `@tuvren/sdk` at all. `grep` across `typescript/host/repl/src` finds 14 files importing from `@tuvren/runtime` (`repl-host.ts`, `repl-provider.ts`, `repl-shell.ts`, `cli.ts`, and ten more) and zero files importing `@tuvren/sdk`.
  - `@tuvren/sdk` today is exactly the thin helper tier ADR-037 designed: payload codecs (`createAesGcmPayloadCodec`, `createIdentityPayloadCodec`, `IDENTITY_PAYLOAD_CODEC`) and schema-authoring helpers (`defineTool`, `asSchema`, `jsonSchema`, `zodSchema`, `standardSchema`). Its `package.json` peer-deps only `@tuvren/core` plus optional `zod`/`@standard-schema/spec`; it has zero backend, runner, or provider dependencies.
  - `@tuvren/runtime`'s `package.json` carries 8 workspace dependencies as regular (non-peer) `dependencies`: `@tuvren/backend-memory`, `@tuvren/backend-postgres`, `@tuvren/backend-sqlite`, `@tuvren/kernel-protocol`, `@tuvren/kernel-runtime`, `@tuvren/mcp-client`, `@tuvren/runner-react`, `@tuvren/telemetry-semconv` — meaning every consumer of `@tuvren/runtime` transitively installs all three backends, the React-oriented runner, and the MCP client whether or not it uses them. This is the widest fan-in of any package in the SDK boundary.
  - Notably, `@tuvren/runtime`'s current `peerDependencies` already lists `@tuvren/sdk` (marked non-optional) — the dependency edge runs the *wrong* direction for the tier split this ADR makes: today `runtime` peer-depends on `sdk`, but the decision below requires `sdk` to depend on `runtime`. This inversion is called out explicitly because it is the concrete migration hazard reviewers will hit first (a naive edit would create a cycle).
  - `runtime/src/index.ts` re-exports `createTuvren` (from `./lib/create-tuvren.ts`) plus curated primitives from `@tuvren/core` subpaths (`Scope`, `DEFAULT_SCOPE`, the `TuvrenError` family, `ApprovalRequest`/`ApprovalResponse` from `@tuvren/core/tools`, event types from `@tuvren/core/events`, execution types from `@tuvren/core/execution`) and the backend/kernel/runner factories. A source audit (`grep` for `ApprovalRequest`, `DEFAULT_SCOPE`, `TuvrenRuntimeError` across `typescript/core/src` and `typescript/runtime/src`) confirms all three already originate in `@tuvren/core` and are merely re-exported by `runtime/src/index.ts` — there is no host-facing contract type that lives *only* in `@tuvren/runtime` today. This narrows what item 4 below needs to move.
  - The 2026-07-04 pre-freeze interview decided: split composition out of the fat engine package, make `@tuvren/sdk` the composition tier, and require hosts to import `{core, sdk, chosen leaf adapters}` — never `@tuvren/runtime` directly. Because ADR-054's freeze has not yet happened (it is explicitly sequenced after ADR-048/049/051), this is exactly the pre-freeze window in which a breaking rearrangement of `createTuvren`'s home and signature is cheap; making it after publication would be a semver-major migration under ADR-054's own precedent.
- **Decision:**
  1. **Move `createTuvren` and the curated re-exports from `@tuvren/runtime` root to `@tuvren/sdk` root.** `@tuvren/sdk` keeps its existing helper exports (`defineTool`, `asSchema`, the payload codecs, etc.) unchanged. `@tuvren/sdk` gains `@tuvren/runtime` as a regular (`dependencies`, not `peerDependencies`) dependency, since `createTuvren`'s implementation needs runtime's internal composition helpers (`createTuvrenRuntime`, the kernel factory wiring). `@tuvren/sdk` continues to peer-dep `@tuvren/core`. `@tuvren/runtime`'s existing `peerDependencies` entry on `@tuvren/sdk` is removed as part of this move — it is the inverted edge described above, and leaving it in place alongside `sdk`'s new dependency on `runtime` would be a cycle.
  2. **`@tuvren/sdk` carries zero backend/runner/provider dependencies.** Consequently ADR-040's kind-tagged string shorthands (`backend: "memory" | "sqlite" | "postgres"`, `driver` defaulting to `"react"`, the `{ kind: "sqlite", options: {...} }` tagged-union forms) are retired. `createTuvren` accepts constructed instances/factories only:
     ```ts
     export interface CreateTuvrenOptions {
       backend: RuntimeBackend;                  // required; no string-kind shorthand, no surprise persistence choice
       runner: RuntimeRunnerFactory;             // required; no implicit "react" default
       provider?: TuvrenProvider;
       tools?: Array<TuvrenToolDefinition | McpToolSource>;
       extensions?: TuvrenExtension[];
       telemetry?: TuvrenTelemetrySink;          // generalized by ADR-058 to the funnel-routing shape
       bounds?: ExecutionBounds;
       kernel?: RuntimeKernel;
       runtimeOptions?: Omit<RuntimeCoreOptions, "kernel" | "runnerRegistry" | "defaultRunnerId">;
     }

     export function createTuvren(options: CreateTuvrenOptions): Promise<TuvrenInstance>;
     ```
     The host installs whichever leaf packages it chose (`@tuvren/backend-sqlite`, `@tuvren/runner-react`, etc. — this is the interview's "hosts pick leaf adapters" contract) and passes constructed instances into one factory call. This keeps `createTuvren` the single batteries-included entrypoint from ADR-040 while removing the transitive install weight the string-kind shorthand implied. This is a pre-freeze breaking change to `CreateTuvrenOptions`, made now precisely because ADR-054's freeze has not happened yet.
  3. **Host import contract (normative).** A host imports `@tuvren/core` (root + subpaths), `@tuvren/sdk`, and its chosen leaf packages (backends, stream adapters, runner, provider bridge, MCP client). A host never imports `@tuvren/runtime` or kernel packages (`@tuvren/kernel-protocol`, `@tuvren/kernel-runtime`) directly. An automated boundary check in the canonical verification path (alongside the existing portability/authority-packet gates) enforces this for the Reference Host and for documentation examples, so the contract cannot silently regress.
  4. **Host-facing contract types stay on `@tuvren/core` subpaths; `@tuvren/runtime` stops re-exporting them on its public surface.** The audit above found no host-facing type that originates only in `@tuvren/runtime` — `ApprovalRequest`/`ApprovalResponse` (`@tuvren/core/tools`), `DEFAULT_SCOPE`/`Scope`/the `TuvrenError` family (`@tuvren/core` root and `@tuvren/core/errors`), and the event/execution types (`@tuvren/core/events`, `@tuvren/core/execution`) already live in `@tuvren/core` per ADR-037's subpath map. This item is therefore a re-export-surface cleanup, not a type migration: `@tuvren/runtime/src/index.ts` removes its curated `@tuvren/core` re-exports (they were only ever a convenience alias), and any consumer importing them through `@tuvren/runtime` switches to importing them from `@tuvren/core`'s subpaths directly, or from `@tuvren/sdk` where `@tuvren/sdk` chooses to re-export them alongside `createTuvren` for ergonomics.
  5. **`@tuvren/runtime` remains published** (`@tuvren/sdk` needs it as a registry dependency) **but is documented and marked internal.** It is excluded from the ADR-054 stable-core guarantee. Its `README.md` and `package.json` `description` state it is an engine package consumed via `@tuvren/sdk`, not a direct host dependency. ADR-054's stable-core enumeration is amended to: `@tuvren/core` (root + subpaths, minus the experimental `@tuvren/core/capabilities` surface per ADR-054/056), `@tuvren/sdk`, the Durable-Read Surface, the `ExecutionHandle`/`awaitResult` surface, `createTuvren` (now the root export of `@tuvren/sdk`), and the published leaf packages (backends, stream adapters, ReAct runner, AI SDK provider bridge, MCP client). `@tuvren/runtime` is not semver-guaranteed and may change shape between minors without triggering ADR-054's breaking-change/migration-window process, since it is no longer a surface any host is meant to depend on directly.
  6. **The Reference Host is re-pointed to the host import contract before the freeze**, not after. All 14 `@tuvren/runtime`-importing files under `typescript/host/repl/src` migrate to importing `createTuvren` and re-exported primitives from `@tuvren/sdk` (or `@tuvren/core` subpaths) and leaf-package instances directly; `typescript/host/repl/package.json` drops its `@tuvren/runtime` dependency and adds `@tuvren/sdk`. This is what lets the eventual freeze audit certify a surface the first-party host actually exercises, rather than certifying a boundary drawn only on paper.
- **Consequences:**
  - Consumer install weight drops for any host that only needs, say, the SQLite backend and one runner: it no longer force-installs all three backends plus `@tuvren/runner-react` plus `@tuvren/mcp-client` merely by depending on the composition entrypoint.
  - ADR-037's peer-dependency single-instance guarantee (version-skew safety for `@tuvren/core`) is preserved unchanged: leaf packages still peer-dep `@tuvren/core` with a tilde range at publish time, and `@tuvren/sdk` continues to peer-dep `@tuvren/core` rather than bundling it.
  - The batteries-included ergonomic promise from ADR-040 shifts from "one import, string-kind shorthand" to "one factory call, explicit adapter imports" — an accepted trade recorded in the 2026-07-04 interview. Hosts write one extra import line per adapter they use but never pay for adapters they don't.
  - Conformance plans that exercise `createTuvren` (including the `runtime-api-batteries-included` check set from ADR-040) move to target the `@tuvren/sdk` surface instead of `@tuvren/runtime`; their assertions on `CreateTuvrenOptions` are updated to reflect the instances-only shape (no string-kind cases to cover).
  - The freeze gate under ADR-054/056 snapshots `@tuvren/sdk` + `@tuvren/core`, not `@tuvren/runtime`; a future breaking change inside `@tuvren/runtime` (short of a change that also breaks `@tuvren/sdk`'s public contract) does not require a semver-major or migration window.
  - Fixing the current `runtime → sdk` peer-dependency inversion requires a coordinated single-commit change (remove the peer dep from `runtime`, add the regular dep from `sdk` to `runtime`, move `create-tuvren.ts` and its lib helpers) to avoid an intermediate state where the two packages form a dependency cycle; this is called out for whoever executes the migration epic.
  - Documentation, examples, and the Reference Host all reference `@tuvren/sdk` as the entrypoint going forward; any remaining `@tuvren/runtime` import outside of `@tuvren/sdk`'s own internals or the workspace's own build tooling is a boundary violation the automated check in item 3 catches.
