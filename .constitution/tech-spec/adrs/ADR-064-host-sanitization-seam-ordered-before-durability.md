### ADR-064 Host Sanitization Seam Ordered Before Durability: One Hook at the Tool-Result Chokepoint

- **Status:** accepted.
- **Context:**
  - ADR-044 made secret isolation an *enforced framework contract* for the surfaces the framework itself owns: provider and MCP credentials are edge-confined, transcript backend options are redacted, telemetry attributes pass an allowlist, and error summaries on the telemetry surface are sanitized runtime text rather than raw transport strings. Every one of those is a case where the framework knows what the secret is because the framework put it there.
  - Tool *result content* is the opposite case, and ADR-044 does not cover it. `ClientReportedResult.content` and a thrown client `err.message` (converted to result content at `typescript/runtime/src/lib/client-endpoint-boundary.ts:150-168`) are host- and peer-authored values whose sensitivity only the host can judge. Today they enter durable lineage unscrubbed, and because kernel history is content-addressed and immutable, that is permanent: a browser-side error that interpolates a token, a cookie, or a signed URL into its message is written into durable state forever. Issue #104 observes that hosts discover this obligation only if they read an inline source comment.
  - The mitigation today is entirely host discipline at the point of endpoint implementation. That is the correct *default* ﻗ the framework genuinely cannot know what counts as a secret in an arbitrary tool payload ﻗ but the framework offers no seam to hang the discipline on, so a host that has a redaction policy has nowhere to install it that is guaranteed to run before durability. Issue #102 makes this urgent rather than theoretical: it is the first change series to run tool content across a real trust boundary, from a browser peer sitting inside a third-party web app, an environment rich in tokens and signed URLs adjacent to every error message.
- **Decision:**
  1. **The framework guarantees a seam and its ordering, never content inspection.** No default redaction, no pattern matching, no key heuristics. A host that installs nothing observes byte-identical behavior to today. This keeps the ADR-044 boundary honest: framework-owned secrets are framework-enforced; host-owned payload policy is host-supplied, and the framework's contribution is the guarantee that host policy runs at a point where it can still matter.
  2. **One hook, at the single durability chokepoint.** `sanitizeToolResult?: (result: ToolResultPart, ctx: { callId, executionClass?, toolName }) => ToolResultPart`, supplied on `AgentConfig` / `RuntimeCoreOptions`, threaded onto `ToolBatchEnvironment`, and applied inside `stageAndEmitResult` (`typescript/runtime/src/lib/tool-execution-helpers.ts`). That function is the sole chokepoint by construction: `emitToolResultEvent` has exactly one call site ﻗ inside it ﻗ and `stageAndEmitResults` delegates to it, so a single application point precedes **both** durable staging and the canonical `tool.result` event. The scrubbed value is therefore what enters kernel history *and* what every stream consumer, projection, and transcript observes; the unscrubbed value never escapes the runtime.
  3. **Symmetric across execution classes by construction, discriminated by context.** Because the hook sits at the shared chokepoint rather than at the client boundary, it covers `tuvren-server`, `tuvren-client`, and provider-mediated results, and every error path ﻗ including the thrown-`err.message` conversion issue #104 specifically names ﻗ with one shape and one ordering rule. A host that wants a policy only for remote-peer content branches on `ctx.executionClass === "tuvren-client"`.
     **`executionClass` is optional, deliberately.** The same chokepoint also terminates `stageImmediateResults`, which carries outcomes decided *before* any binding was resolved ﻗ unknown tool, input-validation failure, policy denial, resume rejection. Those results genuinely have no execution class, and reporting a guessed one (by name lookup against the registry, which misses precisely the unknown-tool case) would be worse than reporting none. It is therefore absent rather than defaulted, and hosts are told the rule directly: a result with no `executionClass` was decided by the framework before dispatch and carries no peer- or provider-authored content, so a policy scoped to a class may safely ignore it ﻗ but a host writing a deny-by-default policy should branch on absence explicitly rather than relying on a negative equality test.
     The alternative considered and rejected was the shape issue #104 literally proposes plus a server-side twin: a `sanitizeResult` on `createClientEndpointBoundary` seeing the raw `ClientReportedResult`, *and* a second hook at the staging path. Two hooks would ship overlapping knobs whose relative ordering hosts must reason about, to buy scrubbing a few call frames earlier on a value that provably cannot reach durable state or the event stream in between.
  4. **Verification is by conformance, not by trusting the hook.** Following ADR-044 ﺡ۶4's precedent that absence of secret material is asserted rather than assumed, tests and a promoted check assert the scrubbed form against **persisted kernel records and captured canonical events**, never against the hook's return value. A test that only asserts the hook was called would pass against an implementation that applied it after staging.
  5. **The obligation is documented where hosts will meet it.** `spec/host/client-endpoint-integration.md` gains a prominent "durable lineage is forever" section stating the permanence, the seam, and its ordering guarantee ﻗ replacing the inline-comment-only discoverability issue #104 reports.
- **Consequences:**
  - Hosts embedding an untrusted or semi-trusted peer gain a supported place to enforce redaction, and the framework can state a real guarantee ﻗ *if you install a policy, it runs before anything becomes permanent* ﻗ instead of documenting an obligation it gives no way to meet.
  - The hook is synchronous and total: it returns a `ToolResultPart` and cannot defer, reject, or fail the call. An async hook would place host-controlled latency on the staging path between a completed tool call and its durable record, in the window where a crash loses the result; a hook that could reject would duplicate the approval seam, which is where "should this proceed" already belongs.
  - **Trade-off recorded:** a host hook that throws is a host defect on a durability-critical path. The runtime does not swallow it into a scrubbed-by-default result ﻗ silently substituting content the host did not author would be a worse failure than a loud one ﻗ so the throw surfaces as the tool call's error. This is stated in the integration guide rather than left to be discovered.
  - ADR-044 is unchanged and remains the authority for framework-enforced secret isolation; this ADR is additive and explicitly scoped to host-authored payload policy.
