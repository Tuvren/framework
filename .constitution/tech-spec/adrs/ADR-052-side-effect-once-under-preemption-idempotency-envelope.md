### ADR-052 Side-Effect-Once Under Preemption via an Idempotency Envelope

- **Status:** accepted; §1 amended by **ADR-065**. The identity is derived from `(turnId, callId)`, not `(runId, callId, fencingToken)`. The original triple could not satisfy this ADR's own "same identity for the same logical call" promise, because a Run is one execution *attempt* (freshly minted per ReAct iteration, per approval resume, and per recovery) and the fencing token rotates on every lease renewal. §2 and §3 are unchanged.
- **Context:** PRD v0.10.0 CAP-P0-068 and Architecture v0.10.0 (Side-Effect-Once Under Preemption Model) require that recovering a stale execution never causes a non-idempotent external side effect to occur twice. ADR-050 removes lease-expiry clock skew, but a residue remains in every multi-worker topology: an external call (a charge, a send, a remote write) can be in flight at the exact instant a worker loses execution authority. Fencing protects the runtime's own database, not the external system. The capability-orchestration work already added the needed metadata: `TuvrenToolDefinition` carries `idempotent?`, `nonRetryable?`, and `maxRetries?` (ADR-046/ADR-047, §3.13/§4.21), the kernel already issues a monotonic `fencingToken` per lease, and recovery already skips completed tool calls by `callId` (`docs/KrakenFrameworkSpecification.md` §4.9). The missing piece is pointing this metadata at the preemption window and propagating an idempotency identity to the systems that actually execute the effect.
- **Decision:** Reuse the existing metadata rather than inventing new fields, and add a derived idempotency envelope:
  1. **Idempotency envelope.** Side-effecting invocations carry an idempotency identity derived from `(runId, callId, fencingToken)`. The Tuvren-server tool path and the Client Endpoint Boundary dispatch envelope both carry it, so a tool implementation or client environment can present it to the external system for deduplication, and a re-dispatch after recovery presents the same identity for the same logical call.
  2. **No retry of in-flight non-idempotent work on authority loss.** When the run lease is lost (renewal fails on a stale fencing token; the handle is already invalidated by `runtime-core-liveness.ts`), the framework must not retry an in-flight invocation marked `nonRetryable` (and `nonRetryable: true` continues to override `idempotent: true` in the retry budget, per ADR-047/Epic BB). Completed, durably-staged results are recovered by `callId` as today; only the in-flight non-idempotent call is abandoned rather than re-run under a dead owner.
  3. **Client-result-as-proposal.** A client-reported result is a proposal; it becomes durable state only through a runtime commit performed under a valid run fencing token, so a stale or late client report (after lease loss or lease expiry) can never mutate committed history. This composes the run lease (write authority) with the client-endpoint lease (availability/staleness) without conflating them.
- **Consequences:** Duplicate non-idempotent side effects under preemption are prevented by construction, not by hoping the window is small. No new tool-definition fields are introduced; the envelope is derived from identifiers the kernel and framework already own. `docs/KrakenFrameworkSpecification.md` is amended with the idempotency-under-preemption rule and the client-result-as-proposal invariant. Tool authors that want external dedup must thread the supplied idempotency identity into their external call; tools that do not are unchanged but gain the no-retry-on-authority-loss guarantee for `nonRetryable` work. Conformance for the SaaS-readiness epics proves no duplicated effect and no stale-client commit on recovery.
