### ADR-061 Event-Stream Sequencing and Resumable Cursor: Wire-Level, Anchored to Kernel History

- **Status:** accepted.
- **Context:**
  - Reconnect-without-loss is structurally impossible on the streaming surface today: `TuvrenStreamEvent` carries no sequence number (`BaseEvent` is `timestamp` + optional `source` only), the SSE encoder never populates the frame `id` field so WHATWG `Last-Event-ID` reconnection can never work, and `ExecutionHandle.events()` is documented single-consumer and one-pass. A client that disconnects mid-turn (laptop sleep, tab reload) has no framework-defined way to resume the stream from a position; every host would invent buffer code.
  - Resumability is Tuvren's thesis: the kernel already owns durable, ordered history, and `state.checkpoint` events already carry `turnNodeHash`. But two constraints bound the design: `state.checkpoint` emission is gated by `enableStateObservability()` and must not be assumed present, and kernel history anchors turn identity without re-projecting fine-grained stream events (`TurnNode.eventHash` is an opaque trigger reference, not an event log) — so replay content can only come from a bounded stream-layer buffer, never from kernel re-projection.
  - Issue #101 names this work; issue #100 (WebSocket transport, ADR-062) is its first wire consumer and FastBlocks the first host consumer. `TurnStartEvent.resumedFrom` is a distinct, pre-existing concept — kernel-level resumption of a paused Turn's *execution* — and is deliberately untouched: the cursor answers "where in the *event wire stream* should a reconnecting client resume consumption," not "was this Turn's execution resumed."
- **Decision:**
  1. **Sequencing is wire-level, not canonical.** The canonical `TuvrenStreamEvent` union, `BaseEvent` in `spec/streaming/typespec/main.tsp`, the exact-key-set runtime guards, and the ADR-060 session-frame vocabulary all gain nothing. Instead `@tuvren/stream-core` exports a sequencing envelope — `SequencedTuvrenStreamEvent { event, turnId, sequence, cursor }` produced by `createSequencedTuvrenStreamEvents(events)` — that transports carry and in-process consumers never see. The alternative considered, an additive optional `sequence` field on `BaseEvent`, was rejected: it would touch every variant's allowed-key guard, every canonical fixture, and every consumer of the canonical shape, to deliver a field that only wire transports need; and per-transport stamping at serialization time is exactly where "wire-level, additive" belongs.
  2. **The cursor is an opaque token anchored to kernel history when possible.** Encoded shape (authority: `spec/streaming/resume/typespec/main.tsp`, packet `tuvren.framework.event-stream-resume`): base64url JSON `{ v: 1, turnId, turnNodeHash?, sequence }`. `turnId` keys the replay window; `sequence` is the monotonic intra-turn offset (reset to 0 at each `turn.start`); `turnNodeHash` is the durable kernel anchor, present only when state observability emitted a checkpoint, and acts as supplementary validation — a window that no longer retains the cursor's anchor lineage reports out-of-window rather than silently serving events from a different lineage. Malformed tokens and unknown payload versions map to out-of-window, never to a wire error.
  3. **The replay window is bounded and host-owned.** `createReplayBuffer({ capacity })` retains the most recent `capacity` sequenced frames; `replayFrom(cursor)` yields `resumed` (with the retained frames strictly after the position), `out-of-window`, or `unknown-turn`. Both non-resumed statuses mean snapshot fallback: durable kernel state is truth, and the stream layer does not pretend to reconstruct what it evicted. Warm reconnect → replay; cold resume → snapshot. The normative wiring rule — record and live-forward must share one sequencer instance, because two sequencers double-count — is stated in the binding appendix rather than left implicit.
  4. **Transport projections.** SSE gains `toResumableSseFrames`, a new export beside the unchanged `toSseFrames`, which places the cursor in the SSE frame `id` so `Last-Event-ID` works natively; the WebSocket transport (ADR-062) carries the cursor on its own outer envelope and in its handshake. `spec/host/session/` is intentionally not modified — the cursor never enters `SessionEventFrame`.
  5. **Experimental posture.** The packet starts at `0.x` and every new export carries `@experimental` per ADR-056, with the same graduation-on-network-evidence stance as ADR-060 (issue #102 remains the forcing function).
- **Consequences:**
  - Every projection inherits resumability from one primitive: SSE and WS share the sequencer, the cursor codec, and the replay window instead of owning per-transport buffer code, and "your stream is resumable because the kernel remembers, regardless of transport" becomes a framework guarantee with a defined degradation story when observability is off (same-process resume still works; cross-restart resume degrades to snapshot).
  - The canonical stream and the session-frame vocabulary are untouched: no existing consumer, guard, fixture, conformance plan, or language port migrates. Streaming remains TypeScript-only, so no cross-language certification work is created.
  - **Trade-off recorded:** an out-of-window cursor loses events irrecoverably at the stream layer by design — the bounded buffer trades unbounded memory for an honest snapshot fallback, and hosts that need longer replay horizons size `capacity` accordingly rather than the framework guessing.
