# Stage 1 (PRD) Changelog

Local SemVer history for `.constitution/prd/`.

- v0.12.0 - **Two-funnel data model + host-facing surface hardening (post-audit interview, session 2026-07-04).** Added the two-funnel commitment to the Operational Observability and Telemetry epic: CAP-P0-071 (session content and operational telemetry are two distinct data funnels born at the same runtime execution; only the content funnel is load-bearing for session correctness, continuity, and recovery), CAP-P0-072 (construction-time funnel routing — split destinations, one unified store, or mixed storage substrates — is the host's decision and never a runtime fork), and CAP-P1-073 (ready-made durable telemetry persistence options, additive behind the stable contract). Added the `Content Funnel`, `Telemetry Funnel`, and `Funnel Routing` glossary terms; a `Data Funnel Separation` non-functional constraint (telemetry-destination failure must not fail, block, or corrupt a content-funnel commit); a matching prohibited pattern (no correctness dependency on the telemetry funnel; no implicit cross-funnel commingling); a scope note framing funnel routing as a topology mechanism in the same spirit as the scope seam (mechanism in, destination policy host-owned); a distinction note (content funnel vs telemetry funnel); a new §7.2 Data Funnels conceptual diagram (prior §7.2 Domain Model renumbered to §7.3); vision posture bullets and a success criterion for the split/unified/mixed topologies. Separately hardened the host-facing surface rule at product level: the curated SDK boundary is the *only* supported host-developer surface — internal runtime machinery below it is not a host-facing API even where technically reachable, and the first-party reference host must model what a real downstream host imports (vision posture bullet + Distribution & API Stability constraint sentence). This sharpens existing commitments (CAP-P0-070, the no-private-seam prohibited pattern) rather than adding a new capability; the concrete package boundary belongs to the architecture/tech-spec layers.
- v0.11.0 - **M10.3a rewrite of the PRD layer to the post-#87 physical world (issue #87 §13 M10 DoD, §14 naming).** Resolves the v0.10.1 entry's "stale-pending-M10-rewrite" flag for this layer: no `boundaries/*` physical path was found in any PRD file at re-measurement time (the one `boundaries/*` string in this changelog is the v0.10.1 entry's own historical citation of the promise being resolved here and is kept verbatim). **Terminology:** every execution-strategy "driver" mention (the `Driver Modularity` epic, `execution drivers`, `driver-first baseline`, `driver registry`, the glossary `Driver`/`ReAct Driver` rows, `Framework vs. driver` distinction, and the out-of-scope "Driver hot-swap" entry) is renamed to "runner" (`Runner Modularity`, `execution runners`, `runner-first baseline`, `runner registry`, glossary `Runner`/`ReAct Runner`, `Framework vs. runner`, "Runner hot-swap"), matching the live `typescript/runners/react` (`@tuvren/runner-react`) and `spec/runners/` authority naming landed at 87-M6. No PRD occurrence of "driver" referred to a resource adapter (tty/output, syslog, bus, or provider driver) — that taxonomy branch is an implementation/architecture-layer concern (device and transport adapters) that this product-requirements layer never names by that word, so there is no "stays a driver" residual in this layer. **Product naming (§14):** the product name is re-centered from "Tuvren Runtime" to "Tuvren" throughout (`vision.md`, `glossary.md`, `actors.md`, `capabilities.md`, `domain-model.md`); the glossary's `Tuvren Runtime` row and vision §1.1's "Tuvren is the company brand, Tuvren Runtime is the runtime product, and Kraken is the engine identity" sentence — the two sentences that actually *defined* the old two-name scheme — are rewritten to define the new one-name scheme (Tuvren = brand and product; Kraken = engine identity only, never a name an ordinary library consumer types), with the retired definition preserved as an explicit historical quotation inside the rewritten vision sentence and the glossary's `Do Not Use` column so the naming history stays legible. This is a naming re-centering only: it does not claim the repository itself is already renamed (the physical `Tuvren/runtime` → `Tuvren/framework` repo rename is a separate post-merge step per the epic; current-state prose says "the framework repo," not that the rename already happened). No product capability, scope boundary, or priority changed.
- v0.10.1 - Reviewed for upstream delta; parallel structural migration tracked in GH issue #87 (Tuvren/runtime, "Epic — Restructure into a Sovereign Polyglot Framework"). That epic is explicitly not a `.constitution/` epic (status and milestone tracking live on the GitHub issue itself, per its own §12) and is out of scope for the four-stage planning framework; it is a physical/naming restructure only and does not change product scope, capabilities, or intent. No layer-specific changes required here. M10 of that epic is the point at which `.constitution/` is rewritten to reflect the post-migration repository; any files in this directory citing `boundaries/*` physical paths are stale-pending-M10-rewrite in the interim.
- v0.10.0 - Opened the **SaaS-Readiness program** as the active product block after the production-trust block closed: promoted seven new P0 capabilities (CAP-P0-064 through CAP-P0-070) that let any host embed Tuvren as a multi-tenant SaaS substrate while the product stays an embeddable runtime rather than a hosted service. Added the **Tenancy-Agnostic Scope and Isolation** epic (a scope seam bound at runtime/backend construction; isolation-by-construction via scope-resolved content addressing), the **Data Lifecycle and Erasure** epic (reachability-based reclamation mechanism; crypto-shredding erasure of sensitive untrusted-edge payloads under host-held keys, supporting retention limits and right-to-erasure), the **Execution Sovereignty Under Preemption** epic (side-effect-once recovery, idempotency identity, client-result-as-proposal), the **Conversation-State Ownership** epic (durable lineage is the unconditional source of truth; provider server-side state and continuity artifacts are reconstructable optimizations only — strengthens CAP-P1-031), and the **SDK Stability and Distribution** epic (versioned, registry-published stable SDK contract with an explicitly separated experimental surface). Reframed the GC out-of-scope entry as a mechanism-in/policy-out split (reclamation + erasure are in scope; retention policy stays host-owned), qualified the managed-SaaS and cross-tenant out-of-scope entries to admit the tenancy mechanism, and recorded native first-party provider clients plus provider-stateful dependence as explicit deferrals/exclusions. Added matching non-functional constraints (Tenancy & Isolation, Data Lifecycle & Erasure, Execution Sovereignty, Distribution & API Stability), prohibited patterns, glossary terms (Scope, Isolation-by-Construction, Reclamation, Crypto-Shredding, Erasure, Idempotency Envelope, Conversation-State Ownership), scope/distinction notes, boundary-analysis in-scope items, vision §1.4 roadmap recast with the freeze-after-tenancy+GC sequencing keystone, and five success criteria. Actors and domain-model needed no change (no new persona; concepts extend existing domain language). Implementation is sequenced as a new SaaS-readiness epic block in Tasks, recasting the previously named-but-unticketed productionization roadmap (Epics BE–BI).
- v0.9.1 - Migrated the monolithic `constitution/PRD.md` into the modular `.constitution/prd/` directory (vision, glossary, actors, capabilities, constraints, domain-model, out-of-scope). Content preserved verbatim; no product-scope changes.
- v0.9.0 - Reframed the product as a cross-provider capability orchestration runtime rather than a single tool executor: separated the model-facing Tool Surface from the underlying Capability, recognized four execution classes (provider-native, provider-mediated developer-provided, Tuvren-server, Tuvren-client) with distinct execution/state/credential/observability/control ownership, introduced Bindings, Endpoints, exposure-time and invocation-time Policy, and per-class Observation limits, classified MCP as a binding mechanism rather than an execution class, and recorded the conceptual invariant that every model-visible tool call resolves to a policy-checked capability invocation against a known execution class. Added the Capability Orchestration and Execution Classes epic (CAP-P0-056 through CAP-P1-063), glossary terms, scope distinctions, prohibited patterns, and boundary-analysis items; preserved every existing tool, approval, secret-isolation, and telemetry guarantee (today's developer-defined runtime-executed tool is the Tuvren-server execution class). Implementation is captured as the active Tooling block (Epics AW–BC) in Tasks.md, sequenced ahead of the trust block (Epic BD) and the productionization roadmap (Epics BE–BI).
- v0.8.0 - Promoted production-trust commitments to first-class product scope: durability and crash-recovery guarantees that are verified under fault injection (resume-or-fail-clean with no partial or corrupt lineage), a first-class operational observability and telemetry surface with optional vendor-neutral export, and explicit execution-safety and trust-boundary controls (untrusted MCP and tool inputs, non-bypassable approval enforcement, bounded execution against runaway loops and resource exhaustion, and credential isolation from durable state, telemetry, and transcripts). Recorded the near-term strategic direction (host adoption plus first-party dogfooding, with multi-language parity preserved but deprioritized) and a named post-trust-block roadmap as deferred planning.
