- v0.15.1 - **Idempotency-identity correction in the Client Endpoint Boundary (2026-07-21; issue #102, ADR-065).** Milestone review of v0.15.0 traced its lease-counter reasoning back to a pre-existing defect: the **Client Endpoint Boundary** responsibility has said since v0.10.0 that the dispatch envelope carries an idempotency identity "so a client environment can deduplicate a side effect it re-executes after a reconnect," but the identity was derived from the run fencing token, which rotates on every lease renewal — making it unstable across exactly the reconnects and recoveries it exists to survive. The responsibility now records that the envelope's identity is the *logical call identity* and deliberately excludes the rotating authority epoch, and restates the lease-counter rationale accordingly (its staleness job is bounded to one process; the cross-restart collision it admits concerns the same logical invocation). Also corrected the `flow-remote-session-reattach-and-redelivery.md` §4.22 notes for the same reason, added a note distinguishing the flow's CAP-P0-055 trace (which covers the sanitization step's *placement* on the surfaces CAP-P0-055 defends) from CAP-P0-055's own framework-enforced-credential mechanism, and recorded the dispatch-clock suspension rule that keeps the disconnect grace window from being self-defeating. No logical container, communication style, or trust boundary changed — this is a correction to responsibility prose and one flow.
- v0.15.0 - **Session lifecycle over a droppable link and the result sanitization seam (2026-07-21; issues #102/#104, ADR-063/ADR-064).** Extended the **Host Integration Boundary** responsibility and Inputs with the session-lifecycle seam that sits between the session binding (§v0.13.0) and any carriage binding (§v0.14.0): it holds the session's single outbound claim, owns the one sequencer and one replay window the Event Stream Adapter Layer's wiring rule requires — so numbering can never restart and a shared replay window can never be fed by two independent sequencers — admits at most one live carriage at a time, replays the sequenced event stream from a cursor on reattach, re-presents client invocations still awaiting a result, and converts a lost link or an unresponsive peer into distinct typed capability results rather than a hung invocation. Recorded explicitly that session *identity* (which live session a reconnecting peer is asking for) stays host application state alongside authentication and tenancy, never framework state. Extended the **Client Endpoint Boundary** responsibility with the process-local lease-counter decision (cross-invocation uniqueness comes from call identity; the one cross-restart collision this admits implies the same logical invocation, whose side-effect-once guarantee belongs to the idempotency envelope) and with where its availability signals come from when the endpoint is reached over a droppable link. Extended the **Tool Execution Gateway** responsibility and Inputs with the optional host-supplied sanitization policy applied at result staging: because staging and canonical result emission share one chokepoint, a policy installed there is guaranteed to run before a result becomes durable and before any consumer observes it, for every execution class and every error path — the framework guaranteeing the seam and its ordering only, never content inspection, which stays distinct from the credential isolation the framework does enforce on surfaces it populates itself. Added the new critical flow `flows/flow-remote-session-reattach-and-redelivery.md` (§4.22: single-claim/one-sequencer construction, healthy dispatch and correlated result, link loss with an invocation outstanding branching into grace-window reattach — replay then redelivery then live forwarding, with `session_rejection` frames deliberately not redelivered — versus grace-window expiry into `capability_binding_unavailable` and endpoint detach, the separate connected-but-unresponsive path into `capability_dispatch_timeout`, and the closing durability gate where the sanitization seam runs before staging and before canonical emission), tracing to CAP-P0-019, CAP-P0-061, CAP-P1-063, and CAP-P0-055. Traces to ADR-063 (session-lifecycle ownership, redelivery-not-replay for non-replayable frames, grace window, dispatch timeout, lease-counter durability) and ADR-064 (one sanitization hook at the durability chokepoint, framework owns ordering not policy). No logical container was added or removed and no trust boundary changed — this is a responsibility extension on three existing containers plus one new flow.
- v0.14.0 - **Stream resumability and WebSocket carriage (2026-07-19; issues #100/#101, ADR-061/ADR-062).** Extended the **Event Stream Adapter Layer** responsibility with wire-level stream resumability: it now sequences each event per turn, mints an opaque resume cursor anchored to durable kernel checkpoint history when observability is on, and retains sequenced frames in a bounded host-owned replay window so a reconnecting transport resumes from the last observed position or falls back to snapshot semantics (durable kernel state as truth, never a stream-layer reconstruction of evicted events) when a position is evicted, its anchor lineage is unretained, or the turn is unknown. Extended the **Host Integration Boundary** responsibility with the WebSocket carriage seam: a WS transport binding may implement the existing duplex session seam (§v0.13.0) over one bidirectional connection, owning handshake, heartbeat, close-code vocabulary, and bounded outbound backpressure only, and carrying the Event Stream Adapter Layer's resume cursor on its handshake and outbound event frames without owning cursor or replay semantics itself. Added the new critical flow `flows/flow-stream-resume-and-ws-transport.md` (§4.21: sequencing and cursor minting against a durable checkpoint anchor, first-connection handshake and event-frame carriage with the cursor riding only event frames, reconnect-with-cursor handshake branching into resumed replay versus snapshot fallback, and connection-level policy — heartbeat liveness, backpressure-triggered close rather than silent frame drop, and unrecognized inbound frames routed to `dispatchInbound` for a typed `session_rejection` rather than a socket close), tracing to CAP-P0-019, CAP-P0-020, and CAP-P1-021 (the Host Control and Streaming Observability epic capabilities for the host control surface, the canonical event stream, and consistent-vocabulary transport parity — a more precise match for a stream-resume-and-carriage flow than the inbound-routing flow's approval/steering/client-capability capability set). Traces to ADR-061 (wire-level sequencing and cursor decision: the canonical `TuvrenStreamEvent` union and the duplex session frame vocabulary are both deliberately unchanged, since sequencing wraps events at the wire and never extends them) and ADR-062 (the WebSocket carriage decision: JSON text envelopes, in-band handshake, and a connection-level close-code vocabulary distinct from the frame-level `session_rejection` vocabulary). No logical container was added or removed, and no existing communication style or trust boundary changed — this is a responsibility extension on the existing Event Stream Adapter Layer and Host Integration Boundary plus one new flow.
- v0.13.0 - **Duplex session seam and remote-client trust boundary (2026-07-18; issue #99, ADR-060).** Extended the **Host Integration Boundary** responsibility, Inputs, and Outputs with the duplex session seam: the framework-defined wire vocabulary through which a remote session peer (a browser tab, mobile client, or any process outside the host process itself) consumes the outbound event stream and submits inbound control frames — client-invocation results, approval responses, steering, and cancellation — that a session binding routes to the held execution handle and the Client Endpoint Boundary. Added the new critical flow `flows/flow-duplex-session-inbound-routing.md` (§4.20: outbound event wrapping, client-invocation dispatch and correlated result with the no-pending-call staleness rejection, approval-response routing through `resolveApproval` with handle-replacement re-bridge and wrong-state rejection, steering, cancellation, and the invariant that every rejected inbound frame gets exactly one typed `session_rejection` echoing its correlation id), tracing to CAP-P0-019, CAP-P1-022, CAP-P0-061, and CAP-P1-063. Added a §2.2 boundary note recording the remote-client trust posture: a remote session peer is untrusted, so every inbound frame is schema-validated before it may touch a handle, invalid or wrong-state frames get typed rejections instead of silent drops, a client-reported result remains a proposal under existing run authority, and the per-dispatch lease token is confined to the session channel rather than broadcast on the canonical event stream. Traces to ADR-060 (the session-frame-only promotion decision: the canonical `TuvrenStreamEvent` union is not extended, so this seam sits entirely above it). No logical container was added or removed, and no kernel surface or existing communication style changed — this is a responsibility extension on the existing Host Integration Boundary plus one new flow and one new boundary note.
